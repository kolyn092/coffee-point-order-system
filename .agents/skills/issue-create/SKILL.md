# 문서 기반 이슈 생성

## 기준

- 이슈 본문 구조는 기능에 `.github/ISSUE_TEMPLATE/feature.md`, 버그에`.github/ISSUE_TEMPLATE/bug.md`를 그대로 사용한다.
- 원 과제 범위는 `docs/PRD.md`의 `원 과제 최소·제출 요구사항` 절, 확정된 제품 정책과 단계는 같은 문서의
  나머지 절을 따른다.
- API 경로·요청·응답·validation·status·에러 코드는 `docs/API.md`, 테이블·제약·인덱스는 `docs/ERD.md`에서만 가져온다.
- 구현·테스트·Git 규칙은 `docs/code-convention.md`를 따른다.
- 필수 문서가 없거나 충돌하면 내용을 지어내지 말고 생성 전에 누락과 필요한 결정을 보고한다.
- 요구사항에 없는 기능과 미래 확장 지점은 이슈로 만들지 않는다.

## 이슈 분해

PRD의 `M<번호>`를 이슈 개수, 제목 또는 분해 단위로 강제하지 않는다. 마일스톤은 범위와 구현 순서를 판단하는
근거로만 사용하고 다음 순서로 이슈를 나눈다.

1. 기반 이슈를 하나 만든다. 기준 문서와 README의 설계 설명, 공통 migration, 로컬 인프라와 애플리케이션 기동처럼
   API 구현 전에 공통으로 필요한 작업만 포함하고 개별 API 구현은 넣지 않는다.
2. 기준 문서에 정의된 API마다 이슈를 정확히 하나 만든다. 각 이슈는 해당 API에 필요한 Entity, Repository,
   Service, Controller와 Test를 포함하는 수직 슬라이스이며 계층별 이슈로 다시 나누지 않는다.
3. 이벤트 발행·소비, Redis 반영처럼 특정 API의 완료에 필요한 부수 작업은 그 API 이슈에 포함한다. 여러 API가
   공유하는 선행 기반 작업만 기반 이슈에 둔다.

모든 API 이슈는 기반 이슈를 `Depends on: #<번호>`로 연결한다. API 이슈끼리는 실제 기술 의존성이 문서에 명시된
경우에만 연결하며, 마일스톤 번호가 연속된다는 이유만으로 직렬 의존성을 만들지 않는다.

제목은 `<검증 가능한 결과 중심의 한국어 제목>`으로 작성한다. 사용자가 요청하거나 저장소에서 별도 추적 규칙으로
정한 경우에만 `[M<번호>]`를 붙인다. 여러 API를 한 이슈에 합치거나 하나의 API를 여러 구현 계층 이슈로 나누지 않는다.

## 작성 규칙

- 목적은 달성할 결과를 한 문장으로 작성한다.
- 포함·제외 범위를 함께 적고 다른 API와 현재 요청 범위 밖의 기능을 제외 항목에 명시한다.
- 기반 이슈에는 README의 ERD·API 명세, 설계 의도, 문제 해결 전략과 기술 선택 이유를 완료 조건으로 포함한다.
- 요구사항마다 `docs/PRD.md`의 ID와 관련 기준 문서 섹션 링크를 붙인다.
- 완료 조건은 경로, HTTP status, 응답 필드, DB 상태, 이벤트, Redis 값 또는 실행 명령으로 검증 가능하게 쓴다.
- 정상 경로뿐 아니라 validation, rollback, 동시성, 정렬·시간 경계와 외부 시스템 실패 조건을 현재 단계만큼 포함한다.
- 테스트 항목은 단위·통합·동시성 중 필요한 종류와 관찰할 최종 상태를 명시한다.
- 문서에 없는 API 경로, 상한, 에러 코드, schema와 설정값을 추측하지 않는다.
- PRD의 현재 요청 범위를 유지하고 후속 단계 기능을 기반 이슈나 API 이슈의 완료 조건에 섞지 않는다.

## 생성 절차

1. `git status`, 현재 브랜치와 원격 저장소를 확인하고 기준 문서가 커밋되어 GitHub에서 열리는지 검증한다.
2. `gh issue list --state all`로 기존·중복 이슈와 선행 단계 상태를 확인한다.
3. 생성 대상마다 제목, 본문, 라벨과 의존 관계 초안을 작성해 사용자에게 한 번에 확인받는다.
4. `gh label list --limit 200`으로 템플릿의 `type:feature` 또는 `type:bug` 라벨을 확인한다.
5. 확인된 초안을 임시 파일에 작성하고 `gh issue create --body-file`로 순서대로 생성한다.
6. 생성 직후 `gh issue view`로 본문, 라벨과 링크를 검증하고 다음 이슈에 실제 번호를 연결한다.

라벨이 없으면 임의 이름으로 대체하지 말고 생성 여부를 확인받는다.
일부 생성이 실패하면 재시도로 중복 이슈를 만들지 말고 현재까지 생성된 번호와 실패 원인을 먼저 확인한다.

## 완료 보고

- 생성한 이슈를 기반 이슈, API 이슈 순서로 번호, 제목과 URL과 함께 보고한다.
- 생성하지 않은 단계, 문서 누락, 보류된 결정과 선행 의존성을 함께 밝힌다.
