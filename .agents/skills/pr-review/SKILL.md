# 프로젝트 코드 리뷰

## 워크플로

1. 가장 가까운 `AGENTS.md`와 리뷰 대상을 확인한다.
    - PR이면 사용 가능한 GitHub 연동을 우선해 실제 base/head, 변경 파일과 전체 diff를 확인한다. 필요할 때 `gh pr diff <번호>`를 사용한다.
    - 로컬이면 `origin/HEAD`와 merge base를 확인하고 staged, unstaged, untracked 변경을 구분한다. 기본 브랜치를 `main`으로 가정하지 않는다.
2. 변경 성격에 맞는 기준 문서를 원문으로 읽는다.
3. diff뿐 아니라 주변 코드, 호출부, 예외 매핑, 테스트, 설정과 migration을 읽는다. 문서와 구현의 불일치 및 현재 브랜치에서 누락된 기준 문서를 숨기지 않는다.
4. 아래 체크리스트로 실제 동작과 실패 경로를 검토한다. 의심되는 문제는 가능한 범위에서 관련 테스트나 정적 확인으로 재현한다.
5. 기본적으로 대화에 리뷰 결과만 보고한다. 사용자가 PR 코멘트나 리뷰 등록을 명시한 경우에만 외부에 게시한다.

## 공통 체크리스트

- [ ] 로직·계약: null, 빈 결과, 경계값, 검증 순서, HTTP status와 에러 코드
- [ ] 호환성: API, schema, Redis 값, 설정 key와 기존 데이터·호출부 영향
- [ ] 예외 처리: 전체 롤백 여부, 삼켜진 예외, 잘못된 fallback과 리소스 누수
- [ ] 보안: 권한, 인젝션, 토큰·비밀번호·개인정보와 실제 접속값 노출
- [ ] 테스트: 성공뿐 아니라 실패 우선순위, rollback, 경합과 장애 경로 검증

## 피드백 형식

- `[P0]` 즉시 중단이 필요한 장애, 금전 데이터 손실 또는 치명적 보안 문제
- `[P1]` 머지 전에 수정해야 하는 재현 가능한 버그, 정합성·동시성·계약 위반
- `[P2]` 현재 동작할 수 있지만 수정 가치가 큰 안정성·테스트·유지보수 문제
- `[P3]` 선택적 개선 또는 사소한 표현 문제

각 지적에는 가능한 범위에서 파일과 줄, 발생 조건, 근거, 영향과 수정 방향을 포함한다. 
수정 코드가 판단을 명확히 할 때만 짧은 예시를 제공한다. 
지적할 문제가 없으면 `발견된 문제 없음`이라고 쓰고 수행하지 못한 검증과 잔여 위험을 함께 적는다. 
실행하지 않은 검증을 통과했다고 표현하지 않는다.
