# 프로젝트 PR 리뷰

## 원칙

- 코드 수정이나 범위 확장 없이 재현 가능한 결함과 검증 공백을 찾는다.
- 현재 PR의 이슈와 마일스톤 범위만 요구한다. P0 PR에 P1·P2 확장 기능을 필수 수정으로 제시하지 않는다.
- 코드가 문서와 다르면 코드를 기준으로 정당화하지 않는다.
- 스타일 취향보다 기능, 데이터 정합성, 동시성, 계약, 장애 처리와 테스트를 우선한다.
- 실행하지 않은 검증을 통과했다고 표현하지 않는다.

## 워크플로

1. 가장 가까운 `AGENTS.md`, 연결 이슈, PR 템플릿과 관련 기준 문서를 원문으로 읽는다.
2. `gh pr view <번호>`로 실제 base/head, 커밋, 연결 이슈, CI 상태와 기존 리뷰를 확인한다.
3. `gh pr diff <번호>`로 전체 diff를 읽고 변경 파일의 주변 코드, 호출부, 설정, migration과 테스트까지 추적한다.
4. PRD 마일스톤과 이슈 완료 조건에 비해 누락·초과된 변경이 없는지 확인한다.
5. 의심되는 문제를 관련 테스트나 정적 확인으로 재현한다. 환경 부족으로 확인하지 못하면 잔여 위험으로 구분한다.
6. 지적을 심각도순으로 정리하고 파일·줄, 발생 조건, 문서 근거, 영향과 최소 수정 방향을 포함한다.
7. 결과를 하나의 `gh pr review --comment --body-file` 코멘트로 게시한 뒤 게시 여부를 확인한다.

PR을 특정할 수 없는 로컬 변경 리뷰라면 외부에 게시하지 않고 대화에만 결과를 보고한다.
이미 같은 내용의 리뷰가 있으면 중복 코멘트를 만들지 않는다.

## 프로젝트 체크리스트

### 공통 계약과 범위

- [ ] 네 API와 한 메뉴 한 잔 주문 범위가 `docs/PRD.md`·`docs/API.md`와 일치한다.
- [ ] 사용자 식별값을 신뢰된 외부 값으로 다루며 P0에 인증·인가를 임의 추가하지 않는다.
- [ ] 입력 오류가 내부 예외, SQL, 비밀값을 응답이나 로그에 노출하지 않는다.
- [ ] 영속 시각과 이벤트 시각, 인기 집계 날짜가 UTC 기준이다.
- [ ] 애플리케이션 인스턴스가 공유 정합성에 영향을 주는 로컬 상태를 보관하지 않는다.

### MySQL과 트랜잭션

- [ ] 메뉴, 포인트와 주문의 단일 원본은 MySQL이다.
- [ ] 같은 사용자의 충전·주문은 포인트 계정 DB 비관적 행 잠금으로 직렬화된다.
- [ ] 주문 저장과 포인트 차감이 한 트랜잭션이며 실패 시 모두 rollback된다.
- [ ] 주문이 결제 당시 메뉴 가격을 저장하고 포인트·금액을 정수로 처리한다.
- [ ] 사용자별 포인트 계정 유일성과 잔액 비음수를 DB 제약으로도 방어한다.
- [ ] `synchronized`, JVM 로컬 락, 로컬 캐시로 다중 인스턴스 정합성을 대신하지 않는다.

### Kafka와 Redis

- [ ] `order.completed`는 주문 commit 후 발행되고 실패가 완료된 주문을 rollback하지 않는다.
- [ ] 이벤트에 주문 ID, 사용자 ID, 메뉴 ID, 결제 금액과 UTC 발생 시각이 포함된다.
- [ ] Redis key, member, score, 8일 만료와 최근 7개 UTC 날짜 합산이 PRD와 일치한다.
- [ ] Top 3는 주문 수 내림차순, 동점이면 메뉴 ID 오름차순이다.
- [ ] Outbox, 게시 재시도, consumer 중복 방지와 Redis fallback은 해당 M5~M8 PR에서만 필수로 검토한다.

### 테스트와 변경 완결성

- [ ] 성공, validation, 빈 결과, 경계값과 실패 우선순위를 검증한다.
- [ ] commit·rollback 후 DB 상태와 Kafka·Redis 부수효과를 함께 검증한다.
- [ ] 동시성 테스트가 실제 경합 시작점, 최종 잔액과 중복 부수효과를 확인한다.
- [ ] MySQL 잠금·제약과 Redis TTL·경합을 H2나 mock 결과만으로 완료 처리하지 않는다.
- [ ] schema 변경에 새 Flyway migration, 엔티티·쿼리, `docs/ERD.md`와 테스트가 함께 변경된다.
- [ ] M0 또는 P0 완료 PR의 README에 설계 문서, 의도, 문제 해결 전략과 기술 선택 이유가 정리된다.

## 심각도

- `[P0]`: 즉시 중단이 필요한 장애, 금전·주문 데이터 손실 또는 치명적 보안 문제
- `[P1]`: 머지 전에 수정해야 하는 재현 가능한 버그, API 계약·정합성·동시성 위반
- `[P2]`: 현재 동작할 수 있으나 장애 대응이나 회귀 방지에 큰 공백이 있는 문제
- `[P3]`: 선택적 유지보수 개선 또는 명확한 컨벤션 위반

## 리뷰 코멘트 형식

1. 지적을 심각도순으로 먼저 작성한다.
2. 각 지적 제목에 `[P#]`를 붙이고 본문에 `경로:줄`, 발생 조건, 영향, 근거와 수정 방향을 적는다.
3. 이어서 실행한 검증과 결과, 실행하지 못한 검증, 문서 누락과 잔여 위험을 적는다.
4. 지적이 없으면 `발견된 문제 없음`이라고 쓰되 검증 범위와 잔여 위험을 생략하지 않는다.
