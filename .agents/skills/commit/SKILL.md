# 변경사항 커밋

## 기본 규칙

- 커밋 제목은 `<type>: <한국어 설명>` 형식을 사용한다.
- `type`은 `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `perf` 중 변경의 주목적 하나를 고른다.
- 제목은 실제 변경 결과를 명령형 또는 간결한 현재형으로 설명한다.
- 브랜치 `feature/<엔드포인트>`의 번호와 GitHub 이슈를 대조하고 번호를 추측하지 않는다.
- 하나의 커밋에는 한 이슈 안에서도 하나의 논리 단위만 포함한다. 사용자 소유의 다른 변경은 스테이징하지 않는다.
- 생성물, 비밀값, 개인 IDE 설정과 로컬 접속값을 커밋하지 않는다.
- 이 스킬에서는 코드나 문서를 수정하지 않는다. 문제가 발견되면 커밋하지 않고 필요한 조치를 보고한다.

## 논리 단위 커밋

- 변경사항을 논리 단위로 나눠 커밋한다. 한 커밋에는 독립적으로 설명·검증·되돌릴 수 있는 하나의 목적만 담는다.
- 파일 종류만 보지 말고 변경 목적과 서로 함께 있어야 동작하는지를 기준으로 나눈다.
- 전체 변경을 한 번에 스테이징하지 말고 아래 기준으로 커밋 계획을 먼저 세운다.

| 변경 조합 | 커밋 기준 |
| --- | --- |
| 기능 A + 기능 B | 기능별 논리 단위로 나눠 각각 커밋한다. |
| 기능 변경 + 그 기능과 무관한 설정 변경 | 기능은 `feat`, 설정은 `chore` 커밋으로 나눈다. |
| `docs/` 변경 + 코드 변경 | 문서를 `docs` 커밋으로 먼저 남기고 코드를 `feat` 또는 해당 목적의 커밋으로 나눈다. |

## 절차

1. 가장 가까운 `AGENTS.md`와 `docs/code-convention.md`의 Git·검증 규칙을 읽는다.
2. `git status --short --branch`, unstaged diff, staged diff와 untracked 파일을 모두 확인한다.
3. 전체 변경을 논리 단위로 나누고 각 단위의 파일, 순서와 커밋 `type`을 정한 계획을 세운다.
4. 현재 브랜치와 `gh issue view <번호>`를 대조해 변경이 이슈 범위와 완료 조건에 맞는지 확인한다.
5. 코드 변경보다 관련 기준 문서가 먼저 또는 함께 반영되었는지 확인한다.
6. 완료 보고에 필요한 테스트를 실행한다. 이전 실행 결과를 재사용한다면 현재 diff 이후 결과인지 확인한다.
7. `git diff --check`를 실행하고 민감정보, 디버그 출력, 빌드 산출물과 무관한 변경을 점검한다.
8. 한 논리 단위의 경로만 명시적으로 스테이징한다. `git add .`나 `git add -A`로 범위를 넓히지 않는다.
9. `git diff --cached --check`, `git diff --cached --stat`, `git diff --cached`로 최종 내용을 다시 확인한다.
10. 형식에 맞는 제목으로 커밋하고 `git show --stat --oneline --decorate HEAD`로 결과를 검증한다.

## 완료 보고

- 커밋 해시와 제목, 포함한 파일 범위, 실행한 검증과 결과를 보고한다.
- 제외한 사용자 변경이나 실행하지 못한 검증이 있으면 함께 밝힌다.
