# 문제 해결 과정 기록

## 원칙

- 기대 동작은 `docs/PRD.md`의 `원 과제 최소·제출 요구사항` 절·제품 정책과 담당 기준 문서에서 가져온다.
- 관찰한 사실, 가설, 추론과 결론을 명시적으로 구분한다.
- 재현하지 못한 문제의 근본 원인을 확정하지 않는다.
- 증거 없이 여러 변경을 한 번에 적용하거나 우연히 사라진 증상을 해결로 기록하지 않는다.
- 진단만 요청받으면 원인과 최소 수정 방향까지만 제시한다. 사용자가 수정을 요청한 경우에만 구현을 변경한다.
- 정책이나 계약 변경이 필요한 해결책은 기준 문서를 먼저 수정하고 사용자 결정을 받는다.
- 운영 데이터 변경, 파괴적 명령과 외부 시스템 조작은 명시적 승인 없이 실행하지 않는다.

## 기록 위치와 상태

- 사용자가 경로를 지정하면 그 경로를 사용한다.
- 지정하지 않으면 `docs/troubleshooting/<YYYY-MM-DD>-<영문-kebab-case>.md`에 기록한다.
- 첫 재현 전에 상태가 `조사 중`인 기록을 만들고 이후 명령, 결과와 판단을 발생 순서대로 추가한다.
- 같은 증상과 원인을 이어서 조사하면 기존 기록에 타임라인을 추가한다.
- 별개의 원인이나 재발 방지 작업이면 새 기록으로 분리하고 관련 문서를 링크한다.

상태는 다음 중 하나만 사용한다.

- `조사 중`: 재현 또는 원인 검증이 진행 중이다.
- `미재현`: 충분한 조건을 확보하지 못해 증상을 재현하지 못했다.
- `원인 확인`: 원인은 입증했지만 해결을 적용하거나 검증하지 않았다.
- `완화`: 증상이나 영향을 줄였지만 근본 원인이 남아 있다.
- `해결`: 원래 재현 절차와 회귀 검증이 모두 통과한다.
- `보류`: 외부 의존성, 권한 또는 사용자 결정이 필요하다.

## 조사 절차

### 1. 증상과 범위 고정

1. `git status --short --branch`로 revision과 사용자 변경을 확인한다.
2. 가장 가까운 `AGENTS.md`, 관련 기준 문서, 이슈와 최근 변경을 읽는다.
3. 기대 결과와 실제 결과를 수치, 상태, 응답 또는 로그로 각각 적는다.
4. 최초 발생 시각과 시간대, 빈도, 영향 대상, 환경과 선행 조건을 기록한다.
5. 비밀값, 개인정보와 실제 인증 정보를 수집하거나 문서에 남기지 않는다.

### 2. 최소 재현

1. 원본 실패를 가장 좁은 테스트, 요청 또는 명령으로 재현한다.
2. 정확한 명령, 입력, fixture, 동시 요청 수, 외부 서비스 상태, 종료 코드와 핵심 출력을 기록한다.
3. 재현 횟수와 성공·실패 횟수를 기록해 간헐성 여부를 구분한다.
4. 시간, 순서, 경합이나 데이터 상태가 원인이 될 수 있으면 제어 가능한 조건으로 고정한다.
5. 재현을 위해 운영 코드를 임시 우회하거나 공유 데이터를 파괴하지 않는다.

재현할 수 없으면 환경 차이와 추가로 필요한 증거를 적고 `미재현`으로 남긴다. 추측한 수정은 적용하지 않는다.

### 3. 가설 검증

- 증상을 설명할 수 있는 가설을 영향도와 검증 비용 기준으로 정렬한다.
- 각 가설에 참일 때 관찰되어야 할 예측과 반증 조건을 먼저 적는다.
- 한 번에 한 변수만 바꾸고 가장 싼 읽기 전용 검증부터 수행한다.
- 명령, 결과와 판정을 조사 타임라인에 즉시 추가한다.
- 반증된 가설을 삭제하지 말고 같은 조사를 반복하지 않도록 남긴다.

| 시각 | 구분 | 가설 또는 작업 | 예측·검증 | 결과 | 판정 |
| --- | --- | --- | --- | --- | --- |
| `<UTC 또는 명시한 시간대>` | 가설 |  |  |  | 대기·채택·기각 |

### 4. 원인 확정

다음 네 요소를 증거로 연결할 수 있을 때만 근본 원인으로 확정한다.

1. 문제를 촉발한 입력, 상태, 경합 또는 외부 실패
2. 잘못된 코드·설정·데이터 조건
3. 실제 증상으로 이어지는 동작 메커니즘
4. 기존 validation, 제약, 테스트 또는 관측이 막지 못한 이유

증상만 가린 변경, 재시작, 데이터 수동 보정과 재시도는 근본 해결과 구분한다.

### 5. 해결과 검증

1. 문서 계약을 유지하는 가장 작은 수정안을 선택한다.
2. 수정 전에 실패하는 회귀 테스트를 가능하면 먼저 추가한다.
3. 원래 재현 절차를 같은 조건으로 다시 실행한다.
4. 관련 단위·통합·동시성 테스트와 전체 회귀 테스트를 필요한 범위만큼 실행한다.
5. DB 상태, 이벤트, Redis와 로그 같은 부수효과가 올바른지 확인한다.
6. `git diff --check`와 실제 diff로 임시 진단 코드, 비밀값과 무관한 변경이 남지 않았는지 확인한다.

원래 증상만 사라지고 회귀 검증을 실행하지 못했으면 `해결`이 아니라 `완화` 또는 `원인 확인`으로 기록한다.
