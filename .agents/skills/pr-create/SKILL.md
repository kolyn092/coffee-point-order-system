# Pull Request 생성

## 기본 규칙

- 기능 PR은 `feature/#<이슈번호>-<슬러그>`에서 `main`으로 생성한다.
- PR 제목은 `[<type>] <한국어 요약>` 형식을 사용한다. `type`은 실제 커밋의 주목적과 맞춘다.
- PR 라벨은 `origin/main..HEAD`의 일반 커밋 제목에서 확인한 모든 `type`을 중복 없이 적용한다. 매핑은 다음과 같다.

| 커밋 type | PR 라벨 |
| --- | --- |
| `feat` | `feature` |
| `fix` | `fix` |
| `refactor` | `refactor` |
| `test` | `test` |
| `docs` | `docs` |
| `chore` | `chore` |
| `perf` | `perf` |

- 병합 커밋은 라벨 산정에서 제외한다. 허용 목록 밖의 커밋 type이나 형식에 맞지 않는 일반 커밋 제목이 있으면 type과 라벨을 추측하지 말고 PR 생성을 중단해 정정이 필요한 커밋을 알린다.
- 적용할 라벨이 GitHub 저장소에 없으면 임의의 대체 라벨을 사용하거나 라벨 없이 PR을 만들지 말고 PR 생성을 중단해 누락 라벨을 알린다.
- PR 본문은 `.github/PULL_REQUEST_TEMPLATE.md`의 모든 섹션을 유지하고 실제 변경으로 채운다.
- 완전히 해결한 이슈만 `Closes #<번호>`로 연결한다. 관련만 있으면 `Related`, 선행 조건이면 `Depends on`에 적는다.
- 이슈 번호, 테스트 결과와 영향 범위를 추측하지 않는다.
- 이 스킬에서는 코드, 테스트와 문서 내용을 수정하거나 새 커밋을 만들지 않는다.

## 절차

1. 가장 가까운 `AGENTS.md`, PR 템플릿과 PR 관련 컨벤션을 읽는다.
2. `git status --short --branch`로 현재 브랜치와 미커밋 변경을 확인한다. 미커밋 변경이 있으면 PR을 만들지 않는다.
3. 원격을 갱신하고 현재 브랜치가 `main`에서 분기했는지, 이슈 번호가 브랜치와 일치하는지 확인한다.
4. `git log origin/main..HEAD`와 `git diff origin/main...HEAD`로 실제 커밋과 전체 변경을 확인한다.
   병합 커밋을 제외한 각 커밋 제목의 type을 확인해 위 매핑에 따라 적용할 PR 라벨 목록을 중복 없이
   산정한다. 형식에 맞지 않거나 허용 목록 밖의 type이 있으면 PR을 만들지 않는다.
5. `gh issue view <번호>`로 완료 조건과 연결 방식을 확인하고 문서·구현·테스트의 일치 여부를 점검한다.
6. `gh label list --limit 200`으로 산정한 모든 PR 라벨이 존재하는지 확인한다. 없는 라벨이 있으면 PR을 만들지 않는다.
7. 현재 diff에 대해 수행된 검증 결과를 확인한다. 필요하면 전체 테스트를 실행하되 결과를 있는 그대로 기록한다.
8. `gh pr list --head <브랜치>`로 기존 PR이 없는지 확인한다.
9. 현재 브랜치를 원격에 push하고 템플릿을 채운 임시 본문 파일로 `gh pr create --base main`을 실행한다. 산정한 각 라벨은 `--label <라벨>` 옵션으로 함께 전달한다.
10. `gh pr view`로 base, head, 제목, 본문, 이슈 링크, 라벨과 URL을 검증한다. 산정한 모든 라벨이 적용되지 않았으면 생성한 PR을 수정하지 말고 차이를 보고한다.

사용자가 명시한 경우에만 draft PR을 만든다. push나 생성이 실패하면 이미 생성된 PR을 먼저 다시 조회해 중복 생성을 방지한다.

## 본문 작성 기준

- `작업 내용`: PRD 마일스톤과 실제 diff의 결과만 요약한다.
- `변경 이유`: 동시성, 트랜잭션, Kafka, Redis 등 이번 변경에 실제 적용된 선택과 근거를 적는다.
- `검증`: 실행한 명령과 성공·실패 결과를 체크박스로 적고 실행하지 않은 항목은 미완료로 남긴다.
- `리뷰 포인트`: 잠금 범위, rollback, 커밋 후 이벤트, UTC 경계, 정렬 등 위험도가 높은 지점을 우선한다.
- `영향 및 남은 위험`: 사용자 동작, migration·설정 영향과 PRD상 다음 마일스톤으로 남긴 한계를 구분한다.

## 완료 보고

- PR 번호, 제목, base/head, 적용한 라벨과 URL을 보고한다.
- 실패하거나 실행하지 않은 검증, 미해결 의존성과 남은 위험을 함께 밝힌다.
