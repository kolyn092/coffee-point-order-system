# 문서 정합성 검사와 동기화

## 원칙

- `docs/PRD.md`의 `원 과제 최소·제출 요구사항` 절을 원 과제 기준선으로 사용한다.
- 같은 파일의 나머지 절을 제품 범위, 단계, 정합성, 동시성, Kafka와 Redis 정책의 기준으로 사용한다.
- 제품 정책은 원 과제 기준선의 필수 요구를 누락하거나 약화할 수 없다. 내부에서 충돌하면 임의로 합치지 않고
  보고한다.
- API 계약은 `docs/API.md`, 데이터 모델과 DB 제약은 `docs/ERD.md`, 구현·테스트 규칙은 가장 가까운
  `AGENTS.md`가 지정한 컨벤션 문서에서 확인한다.
- README는 기준 문서의 요약과 설명이며 새로운 정책의 출처로 사용하지 않는다.
- 코드, 테스트, migration과 설정은 구현 상태의 증거이지 문서 정책의 대체 기준이 아니다.
- 코드에 맞추기 위해 문서를 사후 정당화하지 않는다.
- 검사만 요청받으면 파일을 수정하지 않는다. "동기화해줘", "수정해줘"처럼 변경을 명시한 경우에만 문서를 편집한다.

## 문서 권한

| 내용 | 기준 |
| --- | --- |
| 원 과제의 필수 기능과 제출 요구 | `docs/PRD.md`의 `원 과제 최소·제출 요구사항` 절 |
| 제품 범위, 단계, 동시성·정합성·보안·Kafka·Redis 정책 | `docs/PRD.md` |
| 경로, 요청·응답, validation, HTTP status와 에러 코드 | `docs/API.md` |
| 엔티티, 관계, 컬럼, 타입, 제약과 인덱스 | `docs/ERD.md` |
| 구현, 코드 스타일, 트랜잭션, 테스트, Git과 PR | 가장 가까운 `AGENTS.md`가 지정한 문서 |
| 실제 버전, 의존성과 설정 | 빌드 파일과 `application*.yaml` |

담당 영역이 다른 문서에서 같은 값을 반복하면 담당 기준과 일치하는지 검사한다. 기준 문서 자체가 없거나
`AGENTS.md`의 경로와 실제 파일명이 다르면 비슷한 이름의 파일로 자동 대체하지 말고 깨진 참조로 보고한다.

## 검사 절차

1. `git status --short --branch`로 현재 브랜치와 사용자 변경을 확인한다.
2. 저장소 루트부터 대상 파일까지 적용되는 모든 `AGENTS.md`를 읽는다.
3. `docs/PRD.md`를 처음부터 끝까지 읽고 `원 과제 최소·제출 요구사항` 기준선을 별도로 확인한다.
4. 검사 범위에 연결된 API, ERD, README, 컨벤션, 작업 가이드, migration, 설정과 테스트 문서를 읽는다.
5. 문서 링크와 명시된 파일이 실제로 존재하는지 확인한다.
6. 원 요구사항마다 PRD와 하위 문서의 대응 위치를 연결한 추적 표를 만든다.
7. 같은 개념의 범위, 용어, 수치, 상태, 정렬, 시간 기준, 실패 의미와 단계 배치를 교차 비교한다.
8. 구현 파일은 문서가 실제로 반영되었는지 확인할 때만 사용하고 문서 충돌의 승자를 정하는 근거로 삼지 않는다.
9. 발견 사항을 심각도순으로 정리하고, 수정 모드라면 안전하게 확정할 수 있는 항목만 변경한다.
10. 변경 후 전체 관련 문서를 다시 비교하고 `git diff --check`와 실제 diff를 확인한다.

## 프로젝트 필수 검사 항목

### 원 요구사항 추적

- 메뉴 목록, 포인트 충전, 주문·결제, 최근 7일 인기 메뉴 Top 3의 네 API가 모두 추적되는지 확인한다.
- `1원 = 1P`, 사용자 식별값, 메뉴 ID, 결제 금액과 주문 데이터 전송 요구가 하위 문서에서 유지되는지 확인한다.
- 다중 서버·다중 인스턴스, 동시성, 데이터 일관성과 각 제약의 테스트 요구가 누락되지 않았는지 확인한다.
- README에 ERD, API 명세, 설계 의도, 문제 해결 전략과 기술 선택 이유가 요구되는지 확인한다.

### PRD 내부 정합성

- MVP 포함·제외, P0~P2, M0~M9, Definition of Done과 확장 지점이 같은 기능을 서로 다른 단계에 배치하지
  않는지 확인한다.
- P0 완료 전에 P1·P2 기능을 필수 범위로 끌어오거나, 원 과제의 필수 기능을 후속 단계로 미루지 않는지 확인한다.
- MySQL 단일 원본, 포인트 계정 비관적 행 잠금, 주문과 차감의 원자성 및 rollback 정책이 일관적인지 확인한다.
- Kafka 커밋 후 발행의 P0 한계와 P1 Outbox·재시도 정책이 섞이지 않는지 확인한다.
- Redis의 일자별 Sorted Set, 8일 만료, 최근 7개 UTC 날짜, Top 3와 동점 정렬이 모든 문서에서 같은지 확인한다.
- Redis 장애 시 MySQL fallback·재구성 정책이 MVP 설명과 M8 단계 사이에서 모순되지 않는지 확인한다.
- 인증·인가, 여러 메뉴·수량, 취소·환불, 멱등성 같은 제외·확장 기능이 현재 완료 조건으로 유입되지 않는지 확인한다.

### 하위 문서와 구현 증거

- API의 필드, validation, status와 에러 코드가 PRD의 기능·실패 의미를 빠짐없이 표현하는지 확인한다.
- ERD의 유일성, 잔액 비음수, 참조 무결성, 결제 당시 가격과 UTC 저장 정책이 PRD와 일치하는지 확인한다.
- migration, 엔티티, 쿼리와 설정이 담당 문서와 다른 경우 문서 오래됨이 아니라 구현 불일치로 우선 기록한다.
- 빌드 파일의 Java, Spring Boot, MySQL, Redis와 Kafka 관련 실제 설정이 문서에 적힌 값과 일치하는지 확인한다.
- 테스트가 정상, validation, rollback, 경합, 정렬과 시간 경계를 현재 단계만큼 검증하는지 확인한다.

## 발견 사항 분류

| 심각도 | 기준 |
| --- | --- |
| 차단 | 필수 요구 누락, 기준 문서 충돌, 정책 결정 없이는 안전하게 수정할 수 없는 문제 |
| 중요 | API·DB·정합성·단계 계약 불일치, 존재하지 않는 필수 문서나 깨진 기준 링크 |
| 경미 | 의미를 바꾸지 않는 용어, 중복 설명, 링크와 표현의 부분적 불일치 |

유형은 `충돌`, `누락`, `모호`, `범위 누출`, `깨진 참조`, `구현 불일치`로 구분한다. 각 발견 사항에
`경로:줄`, 서로 다른 두 근거, 영향, 담당 기준 문서와 최소 수정 방향을 포함한다.

## 동기화 규칙

- 오탈자, 깨진 내부 링크와 확정된 용어처럼 의미가 바뀌지 않는 항목만 기계적으로 수정한다.
- 제품 범위, API 계약, DB 제약, 동시성 방식, 실패 의미나 마일스톤이 달라지는 변경은 설계 결정으로 취급한다.
- 설계 방향이 갈리면 추천안을 첫 번째에 둔 장단점 비교표를 제시하고 사용자 결정을 받은 뒤 기준 문서부터 수정한다.
- 기준 문서를 먼저 바꾸고 파생 문서, 코드와 테스트 순서로 반영한다.
- PRD의 `원 과제 최소·제출 요구사항`을 이후 제품 정책에 맞게 삭제하거나 완화하지 않는다.
- 현재 요청 범위를 넘어서는 기능을 정합성 수정이라는 이름으로 추가하지 않는다.
- 사용자 소유의 기존 변경과 무관한 문서를 함께 정리하지 않는다.

## 결과 형식

다음 순서로 결과를 작성한다.

1. 검사 범위와 사용한 기준 문서
2. 요구사항 추적 표와 누락 상태
3. 심각도순 발견 사항 또는 `발견된 불일치 없음`
4. 수정한 파일과 변경 이유
5. 실행한 검증과 결과
6. 미확정 결정, 누락 문서와 잔여 위험

검사하지 않은 문서나 실행하지 않은 검증을 완료했다고 표현하지 않는다.
