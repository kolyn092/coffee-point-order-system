# 이슈 작업

## 작업 원칙

- 저장소의 가장 가까운 `AGENTS.md`와 그 문서가 지정한 기준 문서를 먼저 읽는다.
- 원 과제 요구사항과 프로젝트의 확정 범위와 정책은 `docs/PRD.md`를 따른다.
- API 계약은 `docs/API.md`, DB 모델은 `docs/ERD.md`, 구현과 검증 규칙은 `docs/code-convention.md`를 따른다.
- 필수 문서가 없거나 서로 충돌하면 코드를 기준으로 추측하지 말고 차이와 필요한 결정을 보고한다.
- P0 완료 전 P1·P2를 구현하지 않고 `M0 → M1 → ... → M9` 순서를 지킨다.
- 요구사항과 현재 이슈에 없는 확장 기능은 구현하지 않는다.
- 이 스킬에서는 작업 브랜치 생성, 문서·코드·테스트 변경과 검증까지만 수행한다. 커밋과 PR은 별도 스킬에 맡긴다.

## 시작 절차

1. `git status --short --branch`로 현재 브랜치와 사용자 변경을 확인하고 기존 변경을 보존한다.
2. `gh issue view <번호>`로 제목, 본문, 완료 조건, 의존 이슈와 마일스톤을 확인한다.
3. 이슈에 관련된 기준 문서의 해당 섹션과 실제 코드, 테스트, migration, 설정, 의존성을 읽는다.
4. 이슈가 PRD의 어느 마일스톤인지 확인하고 선행 마일스톤이 완료되지 않았다면 작업을 시작하지 않는다.
5. `main`과 원격의 관계를 확인한다. fast-forward가 불가능하거나 사용자 변경과 충돌하면 임의로 정리하지 않는다.
6. `main`에서 `feature/#<이슈번호>-<영문-kebab-case>` 브랜치를 만들거나 같은 이슈의 기존 브랜치를 사용한다.
7. 문서, 코드, 테스트, migration과 설정의 변경 범위를 완료 조건에 연결해 짧은 작업 계획을 세운다.

브랜치 이름은 따옴표로 감싸 실행하고, `feature/` 외의 접두사나 이슈 번호 없는 이름을 만들지 않는다.

## 구현 절차

1. 설계나 계약 변경이 필요하면 기준 문서를 먼저 수정한다.
2. 방향이 갈리면 추천안을 첫 번째에 둔 장단점 비교표로 사용자 결정을 받은 뒤 문서에 확정한다.
3. API 이슈는 Entity부터 Controller까지 계층별로 나누지 않고 하나의 수직 슬라이스로 구현한다.
4. 현재 마일스톤에 필요한 성공, validation, 예외, rollback, 경합과 외부 시스템 실패 경로를 함께 구현한다.
5. 실제 변경과 문서의 완료 조건을 대조하고 불필요한 추상화나 다음 단계 코드를 제거한다.

## 마일스톤별 핵심 범위

- `M0`: 기준 문서·README 설계 설명, Flyway migration, Docker Compose와 MySQL·Redis·Kafka 연결
- `M1`: 메뉴 ID·이름·가격 목록과 ID 오름차순 정렬
- `M2`: 최초 충전 시 포인트 계정 생성, 양의 정수 충전, DB 비관적 잠금과 동시 충전 정합성
- `M3`: 한 메뉴 한 잔 주문, 결제 당시 가격 저장, 차감·주문 원자성, 커밋 후 이벤트 발행과 Mock consumer
- `M4`: Kafka consumer의 일자별 Redis `ZINCRBY`, 8일 만료, 최근 7개 UTC 날짜 Top 3와 동점 정렬
- `M5`: 주문과 Transactional Outbox의 동일 트랜잭션 commit·rollback
- `M6`: Kafka 게시 재시도와 장애 복구 후 최종 발행
- `M7`: consumer 중복 처리와 점수의 단 한 번 증가
- `M8`: Redis 장애 시 MySQL fallback과 Redis 재구성
- `M9`: 부하 테스트, 병목·Kafka lag·전송 실패 관측

## 검증과 완료 보고

- Windows에서는 `.\gradlew.bat test`, 그 외 환경에서는 `./gradlew test`로 전체 테스트를 실행한다.
- MySQL 잠금·제약, Redis TTL·집계처럼 제품별 동작은 mock이나 H2만으로 완료 처리하지 않는다.
- `git diff --check`와 `git status --short`로 공백 오류와 최종 변경 파일을 확인한다.
- 변경 파일, 구현한 완료 조건, 실행한 명령과 결과, 실행하지 못한 검증, 남은 위험을 한국어로 보고한다.
- 실패하거나 실행하지 않은 검증을 통과했다고 표현하지 않는다.
