# Architecture Decision Record 작성과 대체

## 원칙

- ADR에는 결정한 내용보다 결정한 이유, 검토한 대안, 수용한 손해와 다시 검토할 조건을 남긴다.
- 되돌림 비용이 크거나 합리적인 대안이 있어 선택 이유를 잃기 쉬운 기술 결정만 ADR로 만든다.
- 하나의 ADR에는 독립적으로 대체할 수 있는 결정 하나만 기록한다.
- 사용자가 선택하지 않은 설계안을 `채택됨`으로 기록하지 않는다.
- 채택된 ADR은 내용, 상태, 파일명과 경로를 수정·삭제·이름 변경하지 않는다.
- 채택된 결정이 바뀌면 새 ADR을 만들고 새 ADR에서 대체 대상을 명시한다. 이전 ADR에는 `대체됨` 표시도 추가하지 않는다.
- ADR은 결정 당시의 근거를 보존하는 이력 문서다. 현재 제품·API·DB·구현 정책의 기준 문서를 대신하지 않는다.
- 확인할 수 없는 과거의 이유, 결정일과 합의자를 추측하지 않는다. 확인 불가 사실은 그대로 표시한다.
- 사용자가 구현까지 요청하지 않았다면 ADR과 관련 기준 문서만 다루고 코드·설정은 변경하지 않는다.

## ADR 필요성 판정

다음 중 하나 이상에 해당하고 영향이 실질적이면 ADR 후보로 분류한다.

| 판단 기준 | 예시 |
| --- | --- |
| 되돌리려면 데이터 이관, 호환 계층, 중단 또는 이중 운영이 필요함 | 영속 저장소, 식별자, 스키마·이벤트 형식 |
| 여러 인스턴스의 정합성이나 실패 의미를 결정함 | 잠금, 트랜잭션 경계, 메시지 전달 보장, 멱등성 |
| 외부 계약이나 장기간 유지할 기술 종속을 만듦 | 공개 API, Kafka 토픽 계약, 프레임워크·인프라 선택 |
| 보안, 시간, 소유권 또는 복구 경계를 가로질러 영향을 줌 | 인증 경계, UTC 정책, 원본 데이터, 장애 복구 방식 |
| 여러 합리적 대안의 품질 속성 간 교환이 큼 | 정확성 대 처리량, 즉시 일관성 대 최종 일관성 |
| 나중에 같은 논쟁이 반복될 가능성이 높음 | 익숙한 방식과 다른 선택, 의도적으로 받아들인 제약 |

다음 항목은 단독으로 ADR을 만들지 않는다.

- 지역 변수명, 단순 리팩터링과 쉽게 교체할 수 있는 내부 구현 세부사항
- 아직 선택하지 않은 아이디어, 짧은 실험과 폐기 가능한 spike
- 담당 기준 문서의 오탈자나 이미 정해진 API 필드·DB 컬럼을 그대로 옮긴 내용
- 장단점과 장기 영향이 없는 의존성 patch 업데이트

ADR 대상이 아니면 만들지 않는 이유와 대신 수정하거나 참조할 기준 문서를 보고한다.

## 작성 절차

### 1. 기존 결정과 증거 확인

- 기준 문서에 이미 채택된 선택이면 문서, 이슈, 커밋과 테스트에서 당시 근거를 찾는다.
- 근거가 확인되는 범위에서만 소급 ADR을 작성하고 `기록 유형: 소급`을 표시한다.
- 원래 결정일을 확인할 수 없으면 `결정일: 확인 불가`로 적고 현재의 `기록일`과 섞지 않는다.
- 현재 구현만 보고 정책이나 과거 선택 이유를 역으로 만들어내지 않는다.
- 핵심 이유가 남아 있지 않으면 가능한 선택지를 다시 비교하고 사용자의 확인을 받은 뒤 기록한다.

### 2. 선택지 비교와 채택

설계 방향이 갈리면 추천안을 첫 번째에 둔 장단점 비교표를 먼저 제시한다.

| 순서 | 선택지 | 장점 | 단점·실패 위험 | 되돌림 비용 | 마일스톤 적합성 |
| --- | --- | --- | --- | --- | --- |
| 1 | 추천안 |  |  |  |  |
| 2 | 대안 |  |  |  |  |

- 비교 전에 합격 조건과 반드시 지킬 불변 조건을 명시한다.
- 선택지마다 정상 경로뿐 아니라 장애, rollback, 경합, 운영과 이관 영향을 비교한다.
- 추천과 사용자의 최종 선택이 다르면 순서를 사후에 바꾸지 말고 선택 이유를 별도로 기록한다.
- 사용자 선택 또는 명백한 기존 합의가 확인된 뒤에만 `채택됨` ADR을 작성한다.
- 미확정 비교안을 파일로 남겨야 한다는 요청이 있을 때만 `제안됨` ADR을 만든다.

### 3. 파일 생성

- 기본 위치는 `docs/adr/`로 한다. 디렉터리가 없으면 첫 ADR을 만들 때 생성한다.
- 파일명은 `NNNN-<영문-kebab-case>.md` 형식으로 한다. 예: `0001-use-pessimistic-point-lock.md`.
- 기존 파일 번호의 최댓값에 1을 더한 네 자리 번호를 사용한다. 삭제되거나 건너뛴 번호를 재사용하지 않는다.
- 제목은 `# ADR-NNNN: <한국어 결정 제목>` 형식으로 작성한다.
- 관련 문서는 저장소 상대 링크와 문서의 안정적인 제목·요구사항 ID로 연결한다.
- 현재 정책을 바꾸는 결정이면 담당 기준 문서를 먼저 수정하고 같은 변경에서 ADR을 추가한다.
- ADR을 구현보다 먼저 확정한다. ADR 작성 요청만으로 구현 범위를 넓히지 않는다.

`제안됨` ADR은 채택 전까지 수정할 수 있다.
사용자가 채택하면 필요한 내용을 모두 확정한 뒤 상태를 한 번 `채택됨`으로 바꾼다.
그 순간부터 원본 전체를 불변으로 취급한다.

### 4. 기존 ADR 대체

1. 대체할 ADR과 그 ADR이 이미 대체한 선행 ADR을 모두 읽는다.
2. 기존 결정을 바꿔야 하는 새 사실, 제약, 마일스톤 또는 실패 증거를 명시한다.
3. 새 번호로 ADR을 만들고 `대체 대상`에 이전 ADR 링크와 대체 범위를 적는다.
4. 데이터 이관, 호환성, 단계적 전환, rollback과 운영 관측 방법을 결정 결과에 포함한다.
5. 현재 정책의 담당 기준 문서를 새 결정에 맞춘다.
6. 이전 ADR은 상태, 본문, 링크와 파일명을 포함해 어떤 부분도 수정하지 않는다.

여러 ADR의 일부만 대체하면 각각의 번호와 대체 범위를 새 ADR에 명시한다. 대체 관계를 순환시키지 않는다.
현재 결정은 특정 범위에 대해 가장 나중에 채택되어 이전 ADR을 명시적으로 대체한 ADR로 판단한다.
