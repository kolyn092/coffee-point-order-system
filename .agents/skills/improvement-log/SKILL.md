# 개선 근거 기록

## 원칙

- 기능이 변경 전부터 요구사항대로 동작할 때만 개선으로 기록한다.
- 변경 전부터 실패하는 기능, 장애 또는 결함은 개선으로 포장하지 말고 문제 해결 기록으로 전환한다.
- 새로운 기능, API 계약 변경과 제품 정책 변경은 개선 로그만으로 승인하지 않는다.
- 정합성, 안정성 또는 유지보수성 중 실제 목표를 하나 이상 명시하고 측정 방법을 먼저 정한다.
- Before와 After는 같은 명령, 입력, 데이터, 환경과 반복 조건으로 비교한다.
- 실행하지 않은 검증, 재구성한 수치와 주관적 인상을 증거로 쓰지 않는다.
- 사용자가 구현까지 요청한 경우에만 코드나 설정을 변경한다. 기록만 요청받으면 현재 변경과 증거를 분석해 문서만 작성한다.

## 기준 문서와 범위

1. 가장 가까운 `AGENTS.md`와 해당 문서가 지정한 컨벤션을 읽는다.
2. 원 과제 범위는 `docs/PRD.md`의 `원 과제 최소·제출 요구사항` 절에서 확인하고, 제품 정책과 단계는
   같은 문서의 나머지 절에서 확인한다.
3. API·DB 변경 가능성이 있으면 `docs/API.md`와 `docs/ERD.md`를 함께 확인한다.
4. 현재 브랜치, 이슈, diff와 관련 테스트를 확인해 개선 대상을 한 기능 또는 한 위험으로 제한한다.
5. 현재 PRD 단계보다 앞선 Outbox, 재시도, 중복 방지, Redis 복구 같은 기능을 개선 명목으로 도입하지 않는다.

문서와 구현이 충돌하면 구현을 기준으로 개선 목표를 세우지 않는다. 먼저 문서 정합성 문제를 보고한다.

## 기록 위치

- 사용자가 경로를 지정하면 그 경로를 사용한다.
- 지정하지 않으면 `docs/improvements/<YYYY-MM-DD>-<영문-kebab-case>.md`에 기록한다.
- 같은 대상을 이어서 측정하면 기존 기록을 갱신하고, 별개의 개선이면 새 파일을 만든다.
- 기존 파일을 덮어쓰거나 이름 충돌을 임의로 해결하지 말고 날짜, 이슈 번호 또는 범위를 구분한다.

## 진행 절차

### 1. 개선 계약 정의

다음을 코드 변경 전에 확정해 기록한다.

- 대상 기능과 관련 요구사항·PRD ID·마일스톤
- 현재 정상 동작을 보여 주는 검증
- 개선할 위험 또는 구조적 비용
- 변경 후에도 유지할 API, DB 상태, 이벤트와 정렬 등의 불변 조건
- 비교할 지표, 관찰값, 합격 기준과 반복 횟수
- 범위에서 제외할 기능

정책 선택이 필요하면 추천안을 첫 번째에 둔 장단점 비교표로 사용자 결정을 받은 뒤 기준 문서에 먼저 반영한다.

### 2. Before 기준선 수집

1. 기준 revision, 작업 트리 상태, Java·DB·Redis·Kafka 등 관련 환경과 시간을 기록한다.
2. 재현에 필요한 fixture, seed, 동시 요청 수, 반복 횟수와 외부 시스템 상태를 고정한다.
3. 정확한 명령, 입력, 종료 코드와 핵심 결과를 기록한다.
4. 정상 기능의 계약 테스트가 통과하는지 확인한다.
5. 개선 지표를 여러 번 측정해야 하면 개별 값과 요약 방식을 함께 남긴다.

변경이 이미 끝나 Before를 현재 작업 트리에서 실행할 수 없으면 안전한 별도 Git worktree에서 기준 revision을 검증한다.
사용자 변경을 되돌리거나 임의로 stash하지 않는다. 동일한 기준선을 재현할 수 없으면 `비교 불가`로 기록하고
Before/After 개선을 입증했다고 결론 내리지 않는다.

### 3. 변경과 After 수집

1. 요청받은 최소 변경만 적용하고 관련 없는 리팩터링을 섞지 않는다.
2. Before와 같은 조건으로 같은 명령을 실행한다.
3. 불변 조건, 목표 지표와 실패 경로를 모두 확인한다.
4. 전체 회귀 테스트와 프로젝트별 인프라 검증이 필요하면 별도로 실행한다.
5. 결과가 나빠지거나 변동 범위 안이면 개선이라고 표현하지 않는다.

### 4. 비교와 판정

| 관점 | 사용할 수 있는 근거 |
| --- | --- |
| 정합성 | 동시 요청의 최종 잔액·주문 수, rollback 후 DB 상태, 이벤트·Redis 중복 부수효과 |
| 안정성 | 반복 성공률, 실패 주입 결과, timeout·재시도·복구 상태, flaky 재현 횟수 |
| 유지보수성 | 책임·의존 방향, 중복 정책 제거, 변경 지점 수, 테스트 격리와 계약 보존 증거 |

코드 줄 수 감소만으로 유지보수성 개선을 단정하지 않는다. 성능이나 안정성 수치는 충분한 반복과 동일 환경 없이
단정하지 않는다. 정량화할 수 없는 구조 개선은 관찰 가능한 Before/After 코드 사실과 회귀 테스트로 제한해 설명한다.

- 결과:
