## 맥락

P0의 포인트 충전 API는 외부 회원 시스템에서 발급한 `userId`를 받는다. P0 애플리케이션은 회원가입 API를
구현하지 않지만, 충전과 주문은 같은 `point_accounts` 행을 비관적으로 잠가 정합성을 보장해야 한다.

포인트 계정을 충전 시점에 만들면 행이 없는 최초 요청에서 생성 경합과 PK 충돌 처리가 필요하다. 반대로 사용자
등록 시점에 잔액 0의 계정을 만들면 충전은 이미 존재하는 행을 잠그고 잔액만 변경할 수 있다.

## 결정 동인과 불변 조건

- 충전과 주문은 같은 사용자 계정 행을 `SELECT ... FOR UPDATE`로 잠가야 한다.
- 잔액은 `long` 범위의 checked arithmetic으로 계산하고 음수가 될 수 없다.
- P0 애플리케이션의 범위에는 회원가입 API가 포함되지 않는다.
- 포인트 계정의 부재는 정상적인 충전 요청의 상태가 아니라 회원 시스템과의 데이터 정합성 오류로 취급한다.

## 검토한 선택지

| 순서 | 선택지 | 장점 | 단점·실패 위험 | 되돌림 비용 |
| --- | --- | --- | --- | --- |
| 1 | 사용자 등록 시 잔액 0 포인트 계정 사전 생성 | 충전 경로가 잠금 조회와 잔액 변경으로 단순해지고 최초 생성 경합이 없다. | 외부 회원 시스템이 계정을 반드시 만들고 정합성을 유지해야 한다. 계정 부재는 서버 오류가 된다. | 회원 시스템 계약, 기존 사용자 계정 데이터와 충전 오류 의미를 함께 변경해야 한다. |
| 2 | 첫 충전 시 포인트 계정 생성 | 회원 시스템의 사전 생성 보장이 없어도 충전 API가 독립적으로 동작한다. | 생성 경합, PK 충돌과 재시도 정책이 충전 경로에 들어가며 계정 생명주기 소유자가 모호해진다. | 충전 구현과 동시성 테스트를 다시 확장해야 한다. |
| 3 | P0 애플리케이션에 회원가입 API 추가 | 사용자와 포인트 계정 생성의 소유자를 한 애플리케이션에 둘 수 있다. | 인증, 사용자 모델과 회원가입 API가 필요해 P0 범위를 벗어난다. | 제품 범위와 API·DB 모델을 함께 확장해야 한다. |

## 결정

외부 회원 시스템의 사용자 등록 과정에서 `point_accounts` 행을 `balance = 0`으로 사전 생성한다. P0 애플리케이션은
회원가입 API를 만들지 않는다.

충전 API는 계정을 생성하거나 저장하지 않는다. 하나의 트랜잭션에서 사전 생성된 행을 비관적으로 조회하고 엔티티의
잔액만 변경한다. JPA dirty checking이 commit 시 갱신을 반영한다.

포인트 계정이 없으면 회원 시스템과의 정합성 전제가 깨진 것이므로 `INTERNAL_SERVER_ERROR`를 반환한다.

## 결과와 트레이드오프

### 기대 효과

- 충전 코드에서 계정 생성, PK 충돌 처리와 최초 생성 재시도 로직을 제거한다.
- 충전과 주문이 언제나 같은 계정 행을 잠그므로 동시성 경계를 단순하게 유지한다.
- 계정 생명주기 소유자를 사용자 등록 과정으로 명확히 한다.

### 수용한 단점과 위험

- 회원 시스템이 사용자 생성과 포인트 계정 생성을 누락하면 충전이 실패한다.
- 회원 시스템과 포인트 DB 사이의 생성 실패 복구·재조정은 P0 범위 밖이며, 외부 회원 시스템의 책임이다.
- 충전 API만 호출해 새 사용자의 포인트 계정을 만들 수 없다.

## 검증 방법

- MySQL 8.0.16+ Testcontainers에서 사전 생성 계정의 동시 충전 후 최종 잔액을 검증한다.
- 사전 생성 계정의 정상 충전, overflow rollback, 계정 부재의 `INTERNAL_SERVER_ERROR`를 통합 테스트로 검증한다.
- 전체 테스트는 `./gradlew test`로 실행한다.

## 대체 조건

- P0 애플리케이션이 회원가입과 사용자 계정 생명주기를 직접 소유하게 되는 경우
- 외부 회원 시스템이 포인트 계정 사전 생성과 누락 계정 복구를 보장할 수 없는 경우
- 포인트 계정 생성 실패를 사용자에게 별도 공개 오류로 제공해야 하는 경우
