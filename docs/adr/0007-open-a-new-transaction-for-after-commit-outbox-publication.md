# ADR-0007: AFTER_COMMIT Outbox 발행은 별도 트랜잭션으로 처리한다

## 맥락

주문 commit 뒤 실행되는 `AFTER_COMMIT` 리스너는 최초 Outbox 발행을 M6의 행 잠금·상태 확인 경로로 전달한다.
이 시점에는 주문 트랜잭션이 끝났지만 스레드에 연결된 DB 리소스는 남아 있을 수 있다. 기본 전파 방식으로 데이터
접근을 수행하면 종료된 주문 트랜잭션에 참여해 `PUBLISHED` 상태 전이가 commit되지 않을 수 있다.

최초 발행이 Kafka 전송만 성공하고 Outbox를 `PENDING`으로 남기면 다음 재시도 게시자가 같은 payload를 다시
발행한다. 이는 ADR-0006의 단일 게시 소유권과 M7 전 중복 전달 허용 범위를 벗어난다.

## 결정 동인과 불변 조건

- 최초 발행과 재시도는 모두 행 잠금, `PENDING` 상태 확인, Kafka 발행, `PUBLISHED` 전이를 하나의 트랜잭션으로 처리한다.
- 최초 발행 또는 상태 전이 실패가 이미 commit된 주문·포인트 차감을 rollback하지 않는다.
- Kafka 발행 실패와 상태 전이 실패 시 Outbox는 재시도 가능한 `PENDING`으로 남는다.
- 스키마, Outbox 상태 모델, HTTP API와 Kafka 이벤트 형식은 바꾸지 않는다.

## 검토한 선택지

| 순서 | 선택지 | 장점 | 단점·실패 위험 | 되돌리기 비용 |
| --- | --- | --- | --- | --- |
| 1 | 공통 서비스에 `REQUIRES_NEW` 적용 | 공통 잠금·상태 확인을 유지하고 전이를 commit한다. | 발행 실패는 다음 재시도에 의존한다. | 애너테이션과 테스트를 되돌린다. |
| 2 | 전용 서비스에서 새 트랜잭션 시작 | 리스너 경계를 전용 타입으로 드러낸다. | 공통 로직 분기로 경합 정책이 갈라질 수 있다. | 타입과 테스트를 정리한다. |
| 3 | 기본 전파 방식 사용 | 코드 변경이 없다. | `PUBLISHED`가 commit되지 않아 중복 발행한다. | 장애 추적과 재발행 데이터를 정리한다. |

## 결정

`AFTER_COMMIT` 리스너가 호출하는 공통 Outbox 게시 서비스 메서드는 `REQUIRES_NEW`로 별도 트랜잭션을 시작한다.
이 트랜잭션은 `FOR UPDATE SKIP LOCKED` 잠금 획득부터 Kafka 발행과 `PUBLISHED` 전이까지 유지한다. 주기 재시도는
기존과 같이 독립 트랜잭션에서 같은 공통 게시 규칙을 사용한다.

## 결과와 트레이드오프

### 기대 효과

- 정상 주문의 최초 Kafka 발행 뒤 `PUBLISHED`와 UTC `published_at`이 commit된다.
- 뒤이은 스케줄 실행이 같은 Outbox 행을 선택하지 않아 중복 Kafka 레코드를 막는다.
- 최초 발행과 재시도의 DB 행 잠금·실패 처리 규칙이 하나로 유지된다.

### 수용하는 단점과 위험

- 최초 발행 실패는 주문 성공과 분리되어 Outbox를 `PENDING`으로 남기며 다음 주기 재시도에 의존한다.
- Kafka 발행 성공 뒤 DB 상태 전이가 실패하면 M7 전까지 다음 재시도에서 중복 발행할 수 있다.

## 검증 방법

- 주문 commit 뒤 최초 발행이 `PUBLISHED`, UTC `published_at`으로 저장되는지 통합 테스트로 검증한다.
- 최초 발행 뒤 스케줄러를 실행해 Kafka 레코드가 추가로 발행되지 않는지 검증한다.
- Kafka 발행 또는 상태 전이 실패 뒤 Outbox가 `PENDING`, `published_at = NULL`로 유지되는지 검증한다.

## 대체 조건

- Kafka 호출을 행 잠금 밖으로 이동하기 위해 claim·lease 상태 모델을 도입할 때
- M7의 consumer 중복 처리 이후 최초 발행과 재시도의 전달 보장 정책을 다시 정의할 때
