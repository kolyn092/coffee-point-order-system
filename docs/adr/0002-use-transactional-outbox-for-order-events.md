# ADR-0002: 주문 이벤트에 Transactional Outbox를 사용한다

## 맥락

P0 주문은 포인트 차감과 주문 저장을 commit한 뒤 Kafka `order.completed` 이벤트 발행을 한 번 시도한다. 이 방식은
DB commit 뒤 프로세스 또는 Kafka 장애가 발생하면 주문은 완료됐지만 이벤트는 유실될 수 있다.

P1 M5는 주문 성공 사실과 발행할 이벤트를 같은 트랜잭션으로 저장해 이 유실 가능성을 줄여야 한다. 다만 M5의
범위에는 자동 재시도(M6)와 consumer 중복 처리(M7)가 포함되지 않는다. 이미 공개된 주문 API의 성공 의미와 Kafka
이벤트 schema도 바꾸지 않아야 한다.

## 결정 동인과 불변 조건

- 포인트 차감, 주문 저장과 발행 대기 이벤트 저장은 모두 성공하거나 모두 rollback해야 한다.
- 이벤트 payload는 주문 생성 시점의 `order.completed` 계약을 보존해야 한다.
- Kafka 발행 실패는 완료된 주문의 `201 Created` 결과를 실패로 바꾸지 않는다.
- M5는 기존의 커밋 후 Kafka 1회 발행을 유지하고, M6 전에는 자동 재시도를 추가하지 않는다.
- 여러 인스턴스에서 JVM 로컬 상태, `synchronized` 또는 분산 락에 전달 정합성을 의존하지 않는다.
- Kafka 발행 성공 뒤 DB 상태 전이가 실패해 생기는 중복 전달은 M7 consumer 중복 처리 전까지 수용한다.

## 검토한 선택지

| 순서 | 선택지 | 장점 | 단점·실패 위험 | 되돌림 비용 |
| --- | --- | --- | --- | --- |
| 1 | `PENDING` Outbox 저장 후 기존 1회 발행을 유지하고 성공 시 `PUBLISHED` 전이 | 기존 주문 API의 전달 흐름을 유지하면서 commit된 주문마다 재시도 가능한 이벤트 스냅샷을 남긴다. M6이 `PENDING` 행만 이어받을 수 있다. | Kafka 발행 성공 뒤 상태 전이가 실패하면 M6이 같은 이벤트를 다시 발행할 수 있다. | `outbox_events` 데이터 이관, 주문 트랜잭션과 발행 흐름, M6·M7 경계를 함께 바꿔야 한다. |

| 3 | M5에서 즉시 재시도와 consumer 중복 처리까지 함께 구현 | 발행 유실과 중복을 더 빨리 줄일 수 있다. | 재시도 일정, 소유권과 중복 방지 저장소까지 도입해 M5 범위와 마일스톤 순서를 넘는다. | M5·M6·M7의 구현과 테스트를 한 번에 되돌려야 한다. |

## 결정

주문 트랜잭션에서 `outbox_events` 행을 `PENDING` 상태로 함께 저장한다. 행은 주문 ID, Kafka
`order.completed` payload의 JSON 스냅샷, 상태와 UTC 시각을 보관하며, 주문 하나에는 Outbox 이벤트 하나만
허용한다.

commit 후에는 P0의 Kafka 1회 발행을 계속 시도한다. 발행이 성공하면 별도 트랜잭션에서 Outbox 행을
`PUBLISHED`로 전이하고 `published_at`을 기록한다. Kafka 발행 또는 이 상태 전이가 실패하면 행은
`PENDING`으로 남긴다.

M5는 `PENDING` 행을 자동으로 다시 보내지 않는다. M6이 재시도를 구현하고, Kafka 발행 성공 뒤 상태 전이가
실패했을 때 가능한 중복 소비의 부수 효과는 M7에서 막는다.

## 결과와 트레이드오프

### 기대 효과

- commit된 주문마다 발행할 이벤트가 MySQL에 남아 Kafka 장애 뒤에도 유실 대상을 식별할 수 있다.
- 주문·포인트·Outbox의 원자성이 하나의 DB 트랜잭션으로 검증된다.
- M6은 `PENDING` 상태를 재시도 대상으로 사용하고, M7은 수용한 중복 전달 경로를 명확히 검증할 수 있다.
- 이벤트 payload를 생성 시점에 저장하므로 이후 메뉴 가격·이름 변경이 주문 완료 이벤트의 의미를 바꾸지 않는다.

### 수용한 단점과 위험

- Kafka 발행과 `PUBLISHED` 상태 전이는 원자적이지 않다. 발행은 성공했지만 상태 전이가 실패하면 이벤트가
  중복 발행될 수 있다.
- M6 전까지 실패한 이벤트는 `PENDING`으로 보존될 뿐 자동 전달되지 않는다.
- `outbox_events` 테이블과 상태 전이 경로가 추가되어 주문 흐름의 저장·검증 비용이 증가한다.

## 검증 방법

- MySQL Testcontainers 통합 테스트에서 주문·포인트 차감·`PENDING` Outbox 저장의 commit과 rollback을 검증한다.
- Kafka 발행 성공 시 `PUBLISHED`와 `published_at`이 기록되는지 검증한다.
- Kafka 발행 또는 상태 전이 실패 시 주문 API가 성공을 유지하고 Outbox가 `PENDING`으로 남는지 검증한다.
- 같은 사용자 주문 경합 후 성공한 주문마다 Outbox 행이 하나이고 포인트 잔액이 일관적인지 검증한다.

## 대체 조건

- M6에서 1회 직접 발행을 제거하고 별도 게시자가 모든 Outbox 이벤트를 소유하도록 바꿀 때
- Kafka와 DB 사이의 전달 보장 수준 또는 상태 모델을 바꿀 때
- `order.completed` 이외의 이벤트를 같은 Outbox 모델에 저장해야 할 때
- consumer 중복 방지 없이도 중복 발행을 허용할 수 없는 외부 계약으로 바뀔 때
