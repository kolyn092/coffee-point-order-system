# ADR-0008: Redis 원자 스크립트로 인기 메뉴 consumer 중복을 처리한다

- 상태: 채택됨
- 기록 유형: 신규
- 기록일: 2026-07-17
- 결정일: 2026-07-17
- 관련 요구사항: [`docs/PRD.md` POPULAR-01 인기 메뉴 조회](../PRD.md),
  [`docs/PRD.md` P1 M7 Kafka consumer 중복 처리 정책](../PRD.md)
- 관련 마일스톤: M7
- 관련 이슈: [#26](https://github.com/kolyn092/coffee-point-order-system/issues/26)
- 대체 대상: 없음

## 맥락

M6은 Kafka 발행 성공 뒤 Outbox의 `PUBLISHED` 상태 전이가 실패하면 같은 `order.completed` 이벤트를 다시
발행할 수 있다. Kafka의 재전달도 같은 결과를 만들 수 있다. 인기 메뉴 consumer가 이벤트를 받을 때마다 Redis
Sorted Set 점수를 증가시키면, 같은 주문이 여러 번 집계된다.

이 결정은 기존 `order.completed` 이벤트의 필드만 사용해야 한다. 주문 하나에는 Outbox 이벤트 하나가 생성되므로
`orderId`를 중복 판단 기준으로 삼을 수 있다. HTTP 계약과 이벤트 schema는 바꾸지 않으며, 이번 이슈는 정책과
기준 문서만 확정하고 consumer·Redis 처리·migration·테스트 코드는 구현하지 않는다.

## 결정 동인과 불변 조건

- 여러 애플리케이션 인스턴스가 같은 이벤트를 동시에 소비해도 Redis 점수 부수 효과는 한 번만 발생해야 한다.
- 중복 판단은 기존 이벤트의 `orderId`만으로 정의한다.
- 최근 7개 UTC 날짜 집계와 점수 key·처리 표식의 보존 기간이 일관되어야 한다.
- Redis 처리 성공 뒤 Kafka offset commit 전에 중단되어 재전달되어도 점수가 다시 증가하면 안 된다.
- Redis 오류, 데이터 유실과 M8 복구 경계를 명시하되, JVM 로컬 상태·`synchronized`·로컬 캐시에 의존하지 않는다.
- MySQL 소비 기록과 Redis 점수를 하나의 분산 트랜잭션으로 묶지 않는다.

## 결정

사용자 확인에 따라 1번 Redis 원자 스크립트 방식을 채택한다.

consumer는 이벤트 `occurredAt`의 UTC 날짜를 `D`로 계산한다. `D+8 00:00:00Z` 이전에만 다음 두 key를 같은
Redis 스크립트에서 처리한다.

- 처리 표식: `popular:menu:processed:<yyyy-MM-dd>:<orderId>`를 값 `1`로 `SET ... NX PXAT` 한다.
- 점수: 표식 생성에 성공했을 때만 `popular:menu:<yyyy-MM-dd>`에 `ZINCRBY 1 <menuId>`를 실행한다.

두 key의 만료 시각은 모두 `D+8 00:00:00Z`다. 표식이 이미 있으면 스크립트는 중복 결과만 반환하고 점수를 바꾸지
않는다. 같은 이벤트가 여러 consumer 인스턴스에 동시에 전달돼도 Redis 스크립트가 단일 원자성 경계가 된다.

Redis 스크립트 오류는 consumer 실패로 처리해 Kafka offset을 확정하지 않는다. 스크립트가 성공한 뒤 offset commit 전
중단되면 재전달된 이벤트는 남아 있는 표식으로 중복 판정돼 점수를 다시 증가시키지 않는다. `D+8 00:00:00Z` 이후
도착한 이벤트는 현재 최근 7개 UTC 날짜 집계 대상 밖이므로 key를 만들지 않고 소비를 완료한다.

## 결과와 트레이드오프

### 기대 효과

- Kafka 재전달과 M6 재발행이 같은 `orderId`를 전달해도 인기 메뉴 점수는 한 번만 증가한다.
- Redis만으로 다중 인스턴스 간 원자성을 제공하므로 MySQL 소비 기록 테이블, Flyway migration과 상태 회수 정책이
  필요 없다.
- 점수와 표식이 같은 시각에 만료되어 최근 7개 UTC 날짜의 집계 기간 밖 이벤트가 현재 순위에 영향을 주지 않는다.

### 수용한 단점과 실패 경계

- Redis 전체 유실 또는 개별 key eviction으로 표식과 점수가 사라지거나 불일치할 수 있다. 표식만 유실되면 재전달이
  점수를 다시 증가시킬 수 있고, 점수만 유실되면 표식이 재증가를 막아 누락이 남을 수 있다.
- M7은 Redis 유실을 감지·복구하거나 MySQL로 fallback하지 않는다. 유실 데이터의 탐지, MySQL 기준 재구성과
  fallback은 M8의 책임이다.
- `D+8 00:00:00Z` 이후의 늦은 이벤트는 집계하지 않는다. 이 시점에는 해당 날짜가 최근 7개 UTC 날짜 범위 밖이므로
  현재 인기 메뉴 조회에는 영향을 주지 않는다.
- 이 정책은 Redis의 처리 표식 보존 기간 동안의 정확히 한 번인 부수 효과만 보장하며, 장기 소비 감사 이력은 제공하지
  않는다.

## 검증 방법

이번 이슈는 문서 전용 변경이므로 테스트 코드를 추가하지 않는다. M7 구현 시 다음을 Redis 통합 테스트로 검증한다.

- 여러 인스턴스 역할의 consumer가 같은 `orderId`를 동시에 소비해도 일자별 Sorted Set score가 1만 증가한다.
- 스크립트 성공 뒤 offset commit 전 중단을 재현해도 재전달 뒤 score가 추가 증가하지 않는다.
- Redis 스크립트 실패 시 처리 표식과 점수가 함께 남지 않고, Kafka offset이 확정되지 않아 재전달된다.
- 점수 key와 처리 표식이 모두 `D+8 00:00:00Z`에 만료하며, 만료 뒤 같은 이벤트가 현재 7일 집계에 영향을 주지
  않는다.
- Redis 데이터 유실 뒤에는 M7이 자동 복구하지 않고, M8의 MySQL 재구성 경로가 별도로 필요함을 확인한다.

## 대체 조건

- 인기 메뉴 집계보다 긴 소비 이력 보존, 운영 감사 또는 독립적인 재처리 추적이 필요할 때
- Redis 유실을 M8보다 앞선 단계에서 자동 복구하거나, MySQL에 영속적인 소비 상태를 남겨야 할 때
- Redis 스크립트를 실행할 수 없는 배포 구조로 바뀌거나 key의 원자 처리 범위를 유지할 수 없을 때
- 인기 메뉴 집계 기간 또는 만료 정책이 바뀌어 `D+8 00:00:00Z` 보존 규칙이 더는 적합하지 않을 때
