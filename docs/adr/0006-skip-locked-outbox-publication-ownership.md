# ADR-0006: 잠긴 Outbox 이벤트를 건너뛰어 게시 소유권을 정한다

- 상태: 채택됨
- 기록 유형: 신규
- 기록일: 2026-07-17
- 결정일: 2026-07-17
- 관련 요구사항: [`docs/PRD.md` EVENT-01 주문 데이터 전송](../PRD.md#3-기능-요구사항),
  [`docs/PRD.md` P1 M6 Kafka 게시 재시도 정책](../PRD.md#p1-m6-kafka-게시-재시도-정책)
- 관련 마일스톤: M6
- 관련 이슈: [#20](https://github.com/kolyn092/coffee-point-order-system/issues/20)
- 대체 대상: [ADR-0004](0004-retry-pending-outbox-events-with-row-locks.md)의 Outbox 행 잠금 획득과 경합 처리 범위

## 맥락

ADR-0004는 주기 재시도 게시자가 `PENDING` Outbox 행을 잠근 뒤 Kafka에 발행하도록 정했다. 하지만 M5의 주문
commit 후 최초 발행이 완료되기 전에도 스케줄러가 같은 `PENDING` 행을 선택할 수 있었다. 이 경합을 재현한
Testcontainers 통합 테스트에서 최초 발행과 스케줄러가 Kafka 레코드를 각각 한 건씩 발행했다.

최초 발행과 재시도는 같은 Outbox 행에 대해 하나의 게시 소유자만 가져야 한다. 상태 모델은 `PENDING`, `PUBLISHED`
및 `published_at` 조합을 유지하며, M6 범위에 claim·lease 또는 새 상태값을 추가하지 않는다. ADR-0005의 1초
재시도 주기와 5초 Kafka 발행 확인 제한은 그대로 사용한다.

## 결정 동인과 불변 조건

- 최초 발행과 주기 재시도가 같은 `PENDING` 이벤트를 동시에 Kafka에 발행하지 않는다.
- Kafka 발행 또는 상태 전이 실패 시 트랜잭션을 rollback하여 이벤트를 재시도 가능 상태로 남긴다.
- 잠긴 행을 기다리는 동안 이전 `PENDING` 상태를 바탕으로 다시 발행하지 않는다.
- 다중 인스턴스 소유권은 MySQL 행 잠금으로만 정하고 JVM 로컬 상태나 분산 락을 사용하지 않는다.
- Kafka 발행 성공 뒤 DB 상태 전이가 실패해 발생할 수 있는 중복 전달은 M7 전까지 수용한다.
- 기존 스키마, 이벤트 형식, HTTP API와 재시도 주기 설정은 변경하지 않는다.

## 결정

주문 commit 후 최초 발행과 주기 재시도는 모두 `FOR UPDATE SKIP LOCKED`로 대상 Outbox 행을 조회한다. 잠금을
획득한 경로만 `PENDING` 상태를 확인하고 저장된 payload를 Kafka `order.completed`로 발행한 뒤 같은 트랜잭션에서
`PUBLISHED`와 UTC `published_at`으로 전이한다.

다른 게시자가 행을 잠근 경우 조회 결과에서 제외하고 발행하지 않는다. 최초 발행이 행을 건너뛰면 잠금 보유자가
처리를 완료하며, 실패해 `PENDING`으로 rollback되면 ADR-0005의 다음 fixed-delay 주기가 다시 선택한다.

## 결과와 트레이드오프

### 기대 효과

- 최초 발행 지연과 스케줄러 실행이 겹쳐도 이벤트마다 Kafka 레코드는 한 건만 발행된다.
- 잠금 대기 후 이전 상태를 사용해 다시 발행하는 경로를 제거한다.
- 새 상태, migration 또는 분산 락 없이 기존 MySQL 인덱스와 트랜잭션을 사용한다.

### 수용한 단점과 위험

- Kafka 장애 중 잠긴 이벤트는 현재 스케줄 주기에서 건너뛰고 다음 주기에 재시도한다.
- 여러 `PENDING` 행이 있을 때 잠긴 낮은 ID를 건너뛰어 높은 ID가 먼저 발행될 수 있다.
- Kafka 발행 성공 뒤 `PUBLISHED` 전이 실패 시 다음 재시도에서 중복 전달될 수 있다.
- M6에는 잠금 건너뜀 횟수나 `PENDING` 체류 시간의 별도 운영 지표를 추가하지 않는다.

## 검증 방법

- Testcontainers MySQL에서 최초 발행을 지연한 상태로 스케줄러를 실행하고 Kafka 레코드가 한 건인지 검증한다.
- Kafka 발행 실패와 상태 전이 실패 뒤 `PENDING`, `published_at = NULL`이 유지되는지 검증한다.
- 여러 재시도 실행이 경합할 때 한 경로만 발행하고 최종 상태가 `PUBLISHED`인지 검증한다.
- `FOR UPDATE SKIP LOCKED` 쿼리가 MySQL 8.0.16 이상에서 동작하는지 통합 테스트로 검증한다.

## 대체 조건

- 잠긴 낮은 ID 이벤트를 건너뛰는 지연이나 순서 역전이 제품 요구사항을 만족하지 못할 때
- Kafka 호출을 행 잠금 밖으로 옮기기 위해 claim·lease·회수 상태 모델을 도입할 때
- 재시도 횟수, backoff, dead-letter 또는 잠금 건너뜀 관측 정책이 필요할 때
