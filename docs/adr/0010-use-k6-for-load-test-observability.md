# ADR-0010: k6와 두 애플리케이션 인스턴스로 M9 부하를 관측한다

- 상태: 채택됨
- 기록 유형: 신규
- 기록일: 2026-07-19
- 결정일: 2026-07-19
- 관련 요구사항: [`docs/PRD.md` P2 M9 부하 테스트·운영 관측 정책](../PRD.md#p2-m9-부하-테스트와-운영-관측-정책),
  [`docs/PRD.md` §4 정합성과 운영 정책](../PRD.md#4-정합성과-운영-정책)
- 관련 마일스톤: M9

## 맥락

M8까지 MySQL을 원본으로 하는 주문 정합성, Transactional Outbox 재시도, Redis consumer 중복 처리와 MySQL
fallback·재구성이 구현되어 있다. 이 경로는 단일 프로세스의 단위·통합 테스트만으로는 다중 인스턴스 경합, Redis
재구성 중 consumer lag, Kafka 장애 뒤 `PENDING` Outbox의 누적·복구와 실제 요청 지연을 함께 확인하기 어렵다.

M9은 부하 도구·스크립트, 메트릭 수집기, 대시보드와 알림을 구현하지 않는다. 후속 구현 이슈가 같은 환경·부하·판정
기준을 재현할 수 있도록 실행 방식과 원시 결과의 보관 정책을 먼저 정해야 한다.

## 결정 동인과 불변 조건

- 부하 시나리오는 주문·결제, Kafka 게시·소비, 인기 메뉴 조회의 현재 API와 이벤트 계약을 바꾸지 않는다.
- 같은 MySQL·Redis·Kafka를 공유하는 두 애플리케이션에서 DB 행 잠금, Outbox 게시 소유권과 Redis 재구성 경합을
  관측할 수 있어야 한다.
- 부하 결과는 실행 환경의 성능 차이와 무관하게 비교 가능하도록 원시 JSON과 사람이 검토하는 Markdown을 함께 남긴다.
- Kafka 장애와 Redis 유실은 전용 Docker Compose 프로젝트에서만 주입하며, 공유 환경의 데이터·broker·cache를
  조작하지 않는다.
- 정상 시나리오의 정합성, 장애 뒤 `PENDING`과 consumer lag 해소는 성공·실패로 판정한다. 장비 기준이 없는 절대
  지연 SLA는 이번 결정에 추가하지 않는다.

## 결정

사용자 확인에 따라 M9의 부하 발생기는 JavaScript 시나리오를 실행하는 k6 한 개로 한다. k6 실행기는 로컬에 설치한
CLI 또는 버전을 고정한 일회성 컨테이너 중 하나를 사용한다. 컨테이너 방식은 k6 설치를 대체하는 선택지일 뿐,
상시 실행 서비스가 아니다.

k6는 로드밸런서 뒤의 동일한 Spring Boot 애플리케이션 인스턴스 두 개로 요청을 보낸다. 두 인스턴스는 같은 MySQL,
Redis와 Kafka를 공유한다. 일반 혼합 요청과 동일 사용자 경합으로 주문 정합성·lock 병목을, Redis 장애·유실과 Kafka
중단·복구로 fallback·재구성, `PENDING` Outbox와 consumer lag를 관측한다.

각 실행은 `docs/load-test/results/<UTC-실행식별자>/summary.json`에 k6 원본 요약을, 같은 경로의 `report.md`에
실행 환경, 명령, 데이터 준비, 장애 주입 시각, 지표, 판정, 병목 후보와 JSON SHA-256을 기록한다. 같은 Git commit과
환경에서 세 번 실행해 중앙값과 최댓값을 보고한다. 정합성, 장애 뒤 5분 내 `PENDING`·lag 해소는 성공 조건으로,
처리량 증가 둔화와 p95·자원 증가 조합은 병목 후보로 판단한다.

## 결과와 트레이드오프

### 기대 효과

- k6의 JavaScript 시나리오와 JSON 원시 결과를 사용해 부하 조건과 지연·오류 지표를 재현 가능하게 보관한다.
- 두 인스턴스가 동일한 외부 저장소를 공유하므로 단일 인스턴스 시험에서 보이지 않는 경합과 복구 지연을 확인한다.
- Markdown 보고서가 MySQL·Kafka·로그 관측과 원시 k6 결과를 한 실행 단위로 연결해, 대시보드 없이도 결과를 검토할 수
  있다.
- 고정 장비에 종속된 절대 응답시간을 통과 기준으로 오해하지 않고, 정합성·복구와 반복 실행의 변화로 개선 대상을 찾는다.

### 수용한 단점과 위험

- 두 애플리케이션, 로드밸런서와 전용 인프라를 준비해야 하므로 단일 프로세스 부하 실행보다 초기 구성이 크다.
- k6는 Node.js가 아닌 별도 실행기이므로 로컬 설치 또는 일회성 컨테이너 실행이 필요하다.
- 현재 애플리케이션은 전송 실패·Outbox·lag의 전용 메트릭을 제공하지 않으므로, 후속 이슈에서 로그·DB·Kafka 명령의
  5초 관측 수집을 자동화하기 전에는 결과 수집에 운영 절차가 필요하다.
- 5분 복구 기준은 M9의 전용 환경과 정의된 부하에 한정된다. 실제 운영 SLA나 대시보드·알림 정책을 의미하지 않는다.

## 검증 방법

- 후속 구현 이슈에서 k6 스크립트가 PRD의 네 시나리오, VU·기간과 유효 데이터를 그대로 실행하는지 검증한다.
- 일반·경합 시나리오마다 성공 주문·Outbox·포인트 잔액을 MySQL 원본과 대조하고, lag가 0인 뒤 인기 메뉴 API가
  MySQL 집계와 일치하는지 검증한다.
- Kafka와 Redis 장애를 전용 환경에서 주입해 실패 로그·`PENDING`·lag의 증가와 복구 후 5분 내 해소를 검증한다.
- 각 실행의 `summary.json`, `report.md`, JSON SHA-256, commit과 환경 버전이 함께 남는지 검증한다.
- 이 정책 문서 변경은 `git diff --check`로 공백 오류를 확인한다.

## 대체 조건

- 실제 배포 구조가 두 애플리케이션과 로드밸런서를 사용하지 않아 부하 경로를 동일하게 재현할 수 없을 때
- k6가 필요한 프로토콜, 인증 흐름 또는 결과 분석 요구를 충족하지 못해 Gatling 등 다른 도구가 필요한 경우
- 반복 측정에서 MySQL fallback·명명 잠금 또는 Outbox 행 잠금이 허용 가능한 지연·부하를 넘고, 제품 정책 변경이 필요한
  경우
