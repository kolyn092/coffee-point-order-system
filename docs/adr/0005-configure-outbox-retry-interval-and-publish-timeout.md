# ADR-0005: Outbox 재시도 주기와 Kafka 발행 확인 시간을 설정한다

- 상태: 채택됨
- 기록 유형: 신규
- 기록일: 2026-07-17
- 결정일: 2026-07-17
- 관련 요구사항: [`docs/PRD.md` EVENT-01 주문 데이터 전송](../PRD.md#3-기능-요구사항),
  [`docs/PRD.md` P1 M6 Kafka 게시 재시도 정책](../PRD.md#p1-m6-kafka-게시-재시도-정책)
- 관련 마일스톤: M6
- 관련 이슈: [#20](https://github.com/kolyn092/coffee-point-order-system/issues/20)
- 대체 대상: 없음

## 맥락

M6 게시자는 `PENDING` Outbox 이벤트를 주기적으로 재시도해야 한다. 또한 DB 행 잠금을 유지한 트랜잭션 안에서
Kafka 발행 성공을 확인하고 `PUBLISHED`로 전이하므로, 발행 완료를 무기한 기다릴 수 없다. 재시도 주기와 확인
시간은 장애 복구 지연, MySQL 조회 부하, DB 잠금 점유 시간과 중복 전달 가능성에 직접 영향을 준다.

## 결정 동인과 불변 조건

- Kafka 장애가 복구된 뒤 `PENDING` 이벤트가 빠르게 최종 발행되어야 한다.
- 동일 인스턴스에서 실행이 중첩되지 않아야 하며, 인스턴스 간 소유권은 ADR-0004의 DB 행 잠금을 따른다.
- Kafka 완료 확인 실패, 시간 초과와 인터럽트는 모두 Outbox 상태 전이 없이 rollback되어야 한다.
- 시간 초과 뒤 Kafka가 실제로 발행을 완료했을 가능성은 M7 전까지 허용하는 중복 전달 위험이다.
- 재시도 횟수, backoff, 새 상태와 dead-letter 모델은 M6 범위에 추가하지 않는다.

## 검토한 선택지

| 순서 | 선택지 | 장점 | 단점·실패 위험 | 되돌리기 비용 |
| --- | --- | --- | --- | --- |
| 1 | 1초 fixed delay와 5초 Kafka 확인 제한 | 장애 복구 지연을 짧게 유지하고, 같은 인스턴스의 실행 중첩을 막으며 DB 잠금 점유를 유한하게 제한한다. | Kafka가 5초를 넘겨 응답하면 실제 발행 여부와 관계없이 다음 주기에 중복 발행할 수 있다. | 환경 변수로 값을 바꾸고 부하·지연을 다시 검증한다. |
| 2 | 5초 fixed delay와 10초 Kafka 확인 제한 | DB 조회 빈도와 시간 초과를 줄일 수 있다. | 장애 복구 지연과 DB 행 잠금 점유 시간이 늘어난다. | 환경 변수와 검증 기준을 변경한다. |
| 3 | 30초 fixed delay와 30초 Kafka 확인 제한 | 평상시 폴링 부하가 가장 낮다. | 장애 복구가 크게 지연되고 장시간 Kafka 호출이 행 잠금을 점유한다. | 환경 변수와 검증 기준을 변경한다. |

## 결정

각 애플리케이션 인스턴스는 이전 실행 완료 뒤 1초 후 다음 실행을 시작하는 fixed delay 게시자를 사용한다.
저장된 payload의 Kafka 발행은 최대 5초 동안 완료를 기다린다. 발행 실패, 시간 초과, 상태 전이 실패 또는
인터럽트가 발생하면 트랜잭션을 rollback하여 이벤트를 `PENDING`, `published_at = NULL`로 유지한다. 인터럽트는
호출 스레드의 상태를 복원한다.

`OUTBOX_RETRY_FIXED_DELAY`와 `OUTBOX_RETRY_PUBLISH_TIMEOUT_MILLIS` 환경 변수로 값은 조정할 수 있으며, 기본값은
각각 `1s`, `5000ms`다. Kafka producer의 `max.block.ms`도 같은 밀리초 설정을 사용해 메타데이터 대기를 포함한
전체 발행 확인 시간이 5초를 넘지 않도록 한다.

## 결과와 트레이드오프

### 기대 효과

- Kafka 복구 뒤 최대 한 주기 지연으로 보존된 이벤트를 재시도한다.
- Kafka 응답 정지로 인한 DB 행 잠금과 트랜잭션 점유 시간을 5초로 제한한다.
- 설정 변경으로 운영 환경의 부하와 복구 지연을 조정할 수 있다.

### 허용하는 단점과 위험

- 각 인스턴스는 유휴 상태에서도 1초마다 `PENDING` 후보를 조회한다.
- Kafka가 시간 초과 뒤 실제 발행을 완료하면 다음 재시도에서 중복 전달될 수 있다. 이 부수 효과 방지는 M7의 책임이다.
- 긴 Kafka 지연은 반복된 rollback과 재시도를 유발할 수 있다.

## 검증 방법

- Kafka 발행 실패와 시간 초과 뒤 Outbox가 `PENDING`, `published_at = NULL`로 유지되는지 검증한다.
- Kafka가 정상일 때 저장된 payload가 발행되고 UTC `published_at`과 함께 `PUBLISHED`로 전이하는지 검증한다.
- 같은 이벤트를 여러 게시 실행이 경합할 때 한 번만 발행되는지 MySQL 행 잠금으로 검증한다.

## 재검토 조건

- 1초 폴링이 MySQL 부하나 Kafka lag에 유의미한 영향을 준다.
- 5초 확인 제한으로 운영상 과도한 중복 전달 또는 재시도가 발생한다.
- M7 consumer 중복 처리 후 더 강한 전달 보장이나 backoff, dead-letter 정책이 필요해진다.
