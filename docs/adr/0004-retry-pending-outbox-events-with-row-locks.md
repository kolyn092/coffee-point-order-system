# ADR-0004: DB 행 잠금으로 PENDING Outbox 이벤트를 재시도한다

## 맥락

M5는 주문·포인트 차감과 `PENDING` Outbox 저장을 원자화했지만, Kafka 발행 또는 `PUBLISHED` 상태 전이에 실패한
이벤트는 자동으로 다시 보내지 않는다. M6은 Kafka가 복구된 뒤에도 이러한 이벤트를 최종 발행해야 하며, 다중
인스턴스 환경에서 같은 `PENDING` 행을 동시에 게시하지 않아야 한다.

주문 commit 후의 기존 1회 직접 발행은 M5 정책에 따라 유지한다. 따라서 이 결정은 기존 발행을 대체하지 않고,
실패해 남은 `PENDING` 행을 재시도하는 게시자의 소유권과 트랜잭션 경계를 정한다.

## 결정 동인과 불변 조건

- M5가 저장한 `PENDING` payload만 재시도 대상으로 삼는다.
- Kafka 장애 복구 뒤 `PENDING` 이벤트는 최종적으로 발행되고 `PUBLISHED`로 전이되어야 한다.
- 다중 인스턴스 환경에서도 JVM 로컬 상태, `synchronized`와 분산 락에 전달 정합성을 의존하지 않는다.
- `outbox_events`의 상태 모델과 제약은 `PENDING`, `PUBLISHED` 및 `published_at` 조합을 유지한다.
- Kafka 발행과 DB 상태 전이는 원자적이지 않으므로, 발행 성공 뒤 상태 전이가 실패한 중복 전달은 M7 전까지
  수용한다.
- 재시도 횟수, 실패 원인, dead-letter와 consumer 중복 방지는 M6 범위에 추가하지 않는다.

## 결정

각 애플리케이션 인스턴스는 주기적으로 `PENDING` Outbox를 `status`, `id` 순서로 탐색한다. 게시자는 후보 행 하나를
DB 비관적 행 잠금으로 획득하고 상태를 재확인한 뒤, 잠금을 유지한 트랜잭션에서 저장된 payload를 Kafka
`order.completed`로 발행한다.

발행 성공 시 같은 트랜잭션에서 행을 `PUBLISHED`로 전이하고 `published_at`에 UTC 시각을 기록한다. Kafka 발행 또는
상태 전이가 실패하면 트랜잭션을 rollback하여 행을 `PENDING`으로 남기고 다음 주기에 재시도한다.

## 결과와 트레이드오프

### 기대 효과

- Kafka 장애 뒤에도 MySQL에 보존된 이벤트를 다시 게시할 수 있다.
- DB 행 잠금이 동일 이벤트의 동시 게시를 막아 다중 인스턴스에서 재시도 소유권을 정한다.
- 새 상태·이관 없이 M5의 스키마와 인덱스를 사용한다.

### 수용한 단점과 위험

- Kafka 호출 지연만큼 DB 행 잠금과 트랜잭션이 유지된다.
- Kafka 발행 성공 뒤 DB 상태 전이가 실패하면 다음 재시도가 같은 이벤트를 다시 발행할 수 있다.
- M7 전까지 consumer는 중복 이벤트의 부수 효과를 막지 못한다.
- 재시도 간격, 횟수 제한, 실패 원인 기록과 dead-letter는 제공하지 않는다.

## 검증 방법

- Kafka 발행 실패 뒤 Outbox가 `PENDING`, `published_at = NULL`로 남는 통합 테스트를 실행한다.
- Kafka 복구 뒤 다음 게시 실행에서 저장된 payload가 발행되고 행이 `PUBLISHED`와 UTC `published_at`으로 전이하는지
  검증한다.
- 여러 게시 실행이 같은 `PENDING` 행을 경합할 때 최종 DB 상태가 하나의 `PUBLISHED` 행인지 검증한다.
- Kafka 발행 성공 뒤 상태 전이 실패 시 `PENDING`이 유지되어 재시도 대상이 되는지 검증한다.

## 대체 조건

- Kafka 호출을 DB 행 잠금 밖으로 옮기기 위해 claim·lease·회수 상태 모델을 도입할 때
- 재시도 횟수, backoff, 실패 원인 또는 dead-letter 운영 정책이 필요할 때
- M7의 consumer 중복 처리 이후 전달 보장 수준을 강화할 때
