# Outbox 상태·발행 시각 불변식 누락

## 요약

- 상태: 해결
- 영향: `PENDING`인데 발행 시각이 있거나 `PUBLISHED`인데 발행 시각이 없는 행이 저장될 수 있어 재시도 대상과 발행 이력을 신뢰할 수 없었다.

## 기대 결과와 실제 결과

- 기대 결과: `PENDING`은 `published_at IS NULL`, `PUBLISHED`는 `published_at IS NOT NULL`을 DB 제약으로 강제한다.
- 실제 결과: 상태값만 CHECK 제약으로 제한했고 상태와 발행 시각의 조합은 허용됐다.

## 재현 절차

1. `outbox_events`에 `PUBLISHED`와 `published_at = NULL` 조합을 삽입한다.
2. `PENDING`과 `published_at IS NOT NULL` 조합을 삽입한다.

## 수집한 증거

- PR #15 리뷰는 두 잘못된 조합이 DB 직접 변경 또는 이후 배치에서 저장될 수 있음을 지적했다.
- `097d041a4c82ed96e13f16e5fc34fc9840f38a70`은 상태와 발행 시각 CHECK 제약을 보강했다.

## 가설과 검증

- 가설: 애플리케이션 전이 코드만으로 Outbox 상태 모델을 충분히 보호할 수 있다.
  - 검증: DB 직접 변경과 이후 배치는 애플리케이션 전이를 우회할 수 있다.
  - 결론: 기각.

## 근본 원인

상태값의 허용 범위만 DB에서 검증하고, 상태에 따라 필수인 시간값의 조합 불변식을 저장소 계층에 반영하지 않았다.

## 해결 또는 완화

Outbox migration과 ERD에 상태·발행 시각 조합 CHECK 제약을 추가하고 두 잘못된 조합이 거부되는 MySQL 통합 테스트를 추가했다.

## Before/After 검증

- Before: 잘못된 조합의 행이 저장돼 M6 재시도 대상과 발행 이력이 모순될 수 있었다.
- After: PR 재리뷰가 PK·FK·UNIQUE·상태/발행 시각 CHECK와 MySQL 테스트를 확인했다.

## 추가 테스트

- `PENDING` + 발행 시각 존재 거부
- `PUBLISHED` + 발행 시각 없음 거부

## 재발 방지와 문서 반영

- 상태 모델은 enum 값뿐 아니라 상태별 필수·금지 컬럼 조합을 DB 제약으로 표현한다.

## 잔여 위험과 후속 작업

- 전체 Testcontainers 회귀는 당시 제한 시간 내 완료 결과가 없었다. 이후 전체 테스트 시간 초과 기록에서 충분한 제한으로 성공을 확인했다.
