# Redis 서버 시각과 만료 경계 불일치

## 요약

- 상태: 해결
- 영향: 애플리케이션 Clock이 Redis보다 앞서면 아직 유효한 주문 이벤트를 소비 완료해 인기 메뉴 집계에서 누락할 수 있었다.
- 최초 확인 시각과 시간대: 2026-07-18 01:37 KST
- 관련 요구사항: PRD P1 M7의 `D+8 00:00:00Z` 만료, ERD Redis 원자 스크립트
- revision과 환경: PR #30, Redis Lua `TIME`, Kafka consumer

## 기대 결과와 실제 결과

- 기대 결과: Redis key 만료 판단은 점수와 처리 표식을 실제로 저장하는 Redis 서버의 시각을 단일 기준으로 사용한다.
- 실제 결과: Lua 실행 전 애플리케이션 Clock으로 만료를 판정해 Redis는 아직 만료 전이어도 consumer가 offset을 확정할 수 있었다.

## 재현 절차

1. 애플리케이션 Clock을 Redis 서버보다 미래로 설정한다.
2. 이벤트 날짜의 `D+8 00:00:00Z` 직전에 주문 완료 이벤트를 소비한다.

## 수집한 증거

- PR #30 리뷰는 Lua 내부 `TIME`과 애플리케이션 Clock의 불일치로 조기 폐기되는 경로를 지적했다.
- 커밋 `df2520129e680337d9927ad9c4fe578083824554`은 Redis 서버 시각으로 만료 판단을 옮겼다.
- 최종 리뷰는 만료 경계를 Lua 안에서 판단하고 경계 테스트를 확인했다.

## 가설과 검증

- 가설: 애플리케이션 Clock은 Redis 만료 경계의 동등한 시간원이다.
  - 검증: 두 서버의 시각 차이에서 애플리케이션은 만료, Redis는 미만료라는 모순이 생긴다.
  - 결론: 기각.

## 근본 원인

원자 스크립트가 실제 key를 생성·만료시키는 저장소의 시각이 아니라 애플리케이션 인스턴스의 시각을 사전 판단에 사용했다.

## 해결 또는 완화

만료 판단을 Lua 스크립트의 Redis `TIME`으로만 수행해 점수 증가, 처리 표식 저장, 만료 판정을 한 경계에 묶었다.

## Before/After 검증

- Before: 유효한 이벤트가 offset 확정으로 사라져 최근 7일 집계에서 누락될 수 있었다.
- After: Redis 시각이 만료 전이면 애플리케이션 시각과 무관하게 스크립트 결과를 따르는 경계 테스트가 추가됐다.

## 추가 테스트

- 애플리케이션 Clock은 만료 후, Redis 시각은 만료 전인 경우 집계
- Redis 시각이 만료 후인 경우 정상 소비 완료

## 재발 방지와 문서 반영

- 분산 저장소의 TTL·잠금·원자 연산 경계는 해당 저장소가 제공하는 시간원을 사용한다.

## 잔여 위험과 후속 작업

- Redis 장애와 key 유실은 이 시간원 문제와 별개이며 M8 fallback·재구성 정책으로 처리한다.
