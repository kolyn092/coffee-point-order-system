# Redis 상태 key 유실 감지 누락

## 요약

- 상태: 해결
- 영향: 점수 key 또는 상태·점수 key가 유실된 뒤 새 이벤트가 도착하면 부분 집계가 정상 `READY` cache로 보일 수 있었다.
- 최초 확인 시각과 시간대: 2026-07-18 20:19 KST
- 관련 요구사항: PRD P2 M8 Redis fallback·재구성 정책, API 인기 메뉴 fallback 계약
- revision과 환경: PR #34, Redis Lua, MySQL fallback, Kafka consumer

## 기대 결과와 실제 결과

- 기대 결과: 상태 key가 없거나 상태·점수 key 조합이 일치하지 않으면 consumer는 소비를 보류하고,
  조회는 MySQL fallback·재구성으로 완전한 모델을 회복한다.
- 실제 결과: 일부 key가 유실돼도 새 `ZINCRBY`가 빈 Sorted Set을 만들고 `READY`를 다시 기록해,
  유실 전 주문이 빠진 결과를 `200`으로 반환할 수 있었다.

## 재현 절차

1. 특정 UTC 날짜에 주문을 집계한 뒤 점수 key만, 또는 상태·점수 key 모두를 삭제한다.
2. 만료 전 같은 날짜의 새 `order.completed` 이벤트를 소비한다.
3. 인기 메뉴 조회와 consumer offset 확정 여부를 확인한다.

## 수집한 증거

- PR #34 리뷰는 점수 key만 유실된 경우와 상태·점수 key가 함께 유실된 경우를 각각 지적했다.
- `5a7fd3a7317081ea3fc08d6cb358da2988639e81`은 상태 key 누락 시 소비 보류를,
  `69c171d05192f3a691e9a11ef63d14849e1af4d7`은 유실 재구성 검증을 반영했다.
- 최종 리뷰는 상태 key 부재를 score key 존재 여부와 무관하게 불완전으로 분류하는 것을 확인했다.

## 가설과 검증

- 가설: 새 이벤트가 도착해 score key가 다시 생기면 날짜 집계는 완전한 상태다.
  - 검증: Redis는 유실 전 주문을 복구하지 않으므로 새 이벤트만 담긴 score key가 된다.
  - 결론: 기각.

## 근본 원인

consumer 원자 스크립트가 쓰기 전에 상태·점수 key의 완전성을 검증하지 않아,
유실된 파생 조회 모델을 새 이벤트 하나로 정상 상태처럼 표시했다.

## 해결 또는 완화

- 상태 key가 없으면 score key 존재 여부와 무관하게 소비를 보류한다.
- 조회는 MySQL fallback·명명 잠금 재구성으로 완전한 점수·상태·처리 표식을 복원한다.
- consumer는 재구성 표식 동안 offset을 확정하지 않아 재전달로 반영한다.

## Before/After 검증

- Before: 유실 전 주문이 없는 부분 결과가 `200 SUCCESS`로 노출되고 offset이 확정될 수 있었다.
- After: 상태 key 유실과 동시 유실 통합 테스트가 추가됐고, 재리뷰가 fallback·재구성·재전달 흐름을 확인했다.

## 추가 테스트

- `READY` 상태만 남고 score key가 없는 경우
- 상태·score key 모두 없는 경우
- 재구성 중 consumer offset 보류와 이후 중복 방지

## 재발 방지와 문서 반영

- cache의 존재 여부만으로 완전성을 판단하지 않고, 상태 key와 데이터 key의 조합 불변식을 명시한다.

## 잔여 위험과 후속 작업

- fallback 비율, MySQL 집계 시간, 재구성 중 consumer lag는 M9에서 관측한다.
