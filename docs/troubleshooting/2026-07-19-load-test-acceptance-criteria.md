# 장애 복구 부하 테스트 판정 누락

## 요약

- 상태: 해결
- 영향: 장애 복구가 5분을 넘거나 Kafka·Redis 장애 효과가 충분히 재현되지 않아도 부하 테스트가 통과로 기록될 수 있었다.
- 최초 확인 시각과 시간대: 2026-07-19 13:29 KST
- 관련 요구사항: PRD P2 M9 장애 복구·관측·결과 보고서, ADR-0010
- revision과 환경: PR #38, k6, 전용 Docker Compose, MySQL·Redis·Kafka

## 기대 결과와 실제 결과

- 기대 결과: 복구 deadline은 장애 복구 시점부터 5분이고, Kafka 장애 중 `PENDING`·lag 증가,
  Redis 장애·유실 중 MySQL 원본과의 인기 메뉴 일치, 환경·이미지 정보가 모두 판정·보고서에 남는다.
- 실제 결과: deadline이 k6 종료 시점에서 다시 계산됐고, 최종 0만 검사해 장애 중 증가와 부분 성공 응답을 놓칠 수 있었다.
  보고서에는 재현에 필요한 이미지 정보도 부족했다.

## 재현 절차

1. Redis 또는 Kafka를 k6 실행 중간에 복구하고, k6가 복구보다 늦게 끝나게 한다.
2. Kafka 중단 중 `PENDING`·lag가 증가하지 않는 상태를 만든다.
3. Redis 장애 중 부분 인기 메뉴 응답을 반환하게 한 뒤, 복구 후 정상 결과를 반환한다.

## 가설과 검증

- 가설: 시나리오 종료 후 5분 동안 정상 상태면 복구 SLA를 만족한다.
  - 검증: 실제 복구가 중간에 일어나면 복구 뒤 최대 약 6분까지 허용될 수 있다.
  - 결론: 기각.
- 가설: 최종 `PENDING = 0`, lag `= 0`이면 Kafka 장애를 충분히 검증했다.
  - 검증: 장애 중 누적·lag 증가를 전혀 재현하지 않아도 최종값만으로 통과할 수 있다.
  - 결론: 기각.

## 근본 원인

부하 테스트 자동화가 결과의 최종 정상 상태만 확인하고, 장애 주입 시점·구간별 기대 효과·실행 환경을
판정 데이터로 고정하지 않았다.

## 해결 또는 완화

- 장애 복구 epoch에서 5분 deadline을 계산하도록 했다.
- 장애 구간 최대 `PENDING`·lag 증가와 복구 후 해소를 모두 판정한다.
- Redis 장애·유실 구간의 인기 메뉴 응답을 당시 MySQL 집계와 대조한다.
- report에 Compose 이미지와 실행 환경 정보를 남긴다.

## Before/After 검증

- Before: 장애 효과가 없거나 복구 SLA를 초과한 실행이 통과로 기록될 수 있었다.
- After: 최종 리뷰가 복구 기준, Kafka·Redis 구간 검증, 이미지 기록의 반영을 확인했다.
  실제 전용 환경 3회 실행 결과는 아직 이 문서 작성 시점에 없다.

## 추가 테스트

- 장애 복구 시각 기반 deadline 계산
- 장애 구간 `PENDING`·lag 최대값과 복구 후 0
- Redis 장애·유실 구간의 MySQL Top 3 대조
- report의 이미지·Docker·Compose·k6 버전 기록

## 재발 방지와 문서 반영

- 장애 테스트는 최종 상태뿐 아니라 장애가 발생한 증거, 복구 기준 시각, 원본 데이터 대조를 완료 조건에 포함한다.
