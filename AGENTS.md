# AGENTS.md

## 역할

이 파일은 저장소 전체의 작업 진입점이다. 상세 규칙을 중복해서 적지 않고 기준 문서로 연결한다.
더 가까운 하위 `AGENTS.md`가 있으면 해당 문서를 우선한다.

## 작업 원칙

1. **문서 먼저, 코드 나중**: 설계 결정은 구현 전에 `docs/`에 기록한다. 코드는 확정된 문서를 구현한다.
2. **한국어 사용**: 모든 소통과 문서는 한국어로 작성한다. 코드 식별자와 커밋 메시지 제목에만 영어를 허용한다.
3. **최소 조건**: 요구사항에 없는 기능은 만들지 않는다. 향후 확장 가능성은 구현하지 않고 문서의 `확장 지점`에만 기록한다.
4. **선택지 비교 후 결정**: 설계 방향이 갈리면 추천안을 첫 번째에 둔 장단점 비교표를 제시한다. 사용자가 선택한 안을 정책으로 확정하고 문서에 반영한다.

## 필수 절차

1. 요청 범위와 변경 대상을 확인한다.
2. **모든 구현과 리뷰 전에 `docs/code-convention.md` 전체를 읽고 반드시 적용한다.** 컨벤션을 바꿀 때는 이 파일이 아니라 `docs/code-convention.md`를 수정한다.
3. 아래 표의 관련 기준 문서와 작업 범위에 대응하는 `docs/PRD.md` 항목 또는 연결된 이슈를 읽는다.
4. 설계 결정이 필요하면 선택지를 비교해 사용자 결정을 받은 뒤 `docs/`에 먼저 기록한다.
5. 코드, 테스트, migration, 설정과 의존성의 현재 상태를 확인한 뒤 문서에 확정된 범위만 변경한다.
6. `docs/code-convention.md`의 검증 및 완료 보고 규칙을 따른다.

## 개발 워크플로

아래 순서를 지킨다. 각 단계의 상세 규칙은 `.claude/skills/`에 있는 해당 스킬을 단일 기준으로 삼는다.

| 순서 | 단계             | 규칙                                                                |
|---|----------------|-------------------------------------------------------------------|
| 1 | `issue-create` | 기반 이슈를 만들고 API마다 이슈를 하나씩 만든다.                                     |
| 2 | `auto-work`    | Git Flow를 적용해 `main`에서 `feature/#N-슬러그` 브랜치를 만든다.                 |
| 3 | `commit`       | `type: 한국어 설명 (#N)` 형식으로 커밋한다.                                    |
| 4 | `pr-create`    | 기능 PR은 `feature`에서 `main`로 생성한다. |
| 5 | `pr-review`    | 리뷰 결과를 해당 PR의 코멘트로 게시한다.                                          |

## 문서 라우팅

| 확인할 내용                                        | 기준 문서                                                         |
|-----------------------------------------------|---------------------------------------------------------------|
| 원 과제 요구, 제품 범위, 보안·동시성·정합성·Redis 정책             | `docs/PRD.md`                                                 |
| API 요청·응답, validation, status, 에러 코드          | `docs/API.md`                                                 |
| 엔티티, 테이블, 관계, 컬럼, DB 제약·인덱스                   | `docs/ERD.md`                                                 |
| 도메인 규칙, 패키지 구조, 구현, 코드 스타일, 트랜잭션, 테스트, Git·PR | `docs/code-convention.md` |
| 실제 버전·의존성·설정                                  | `build.gradle`, `settings.gradle`, `src/**/application*.yaml` |

충돌 시 담당 영역의 기준 문서를 따르며, 제품·API·DB·성능·컨벤션 순으로 판단한다.
코드가 문서와 다르다는 이유만으로 코드를 기준 삼지 않는다.
모호하거나 기준 문서가 없으면 추측하지 말고 차이와 필요한 결정을 밝힌다.
